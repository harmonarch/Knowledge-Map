# 代数效应

## 一句话定义

代数效应是一种把「发生了什么事」和「这件事应该怎么处理」分开的编程语言机制，可以理解为可恢复执行的 `try / catch`。

## 它解决什么问题

普通异常只能把控制权交给外层 `catch`，原来的调用位置已经回不去了。`async / await` 虽然能表达异步，但会沿着调用链扩散：底层函数一旦变成异步，上层函数也常常要跟着改。

代数效应要解决的是同一类控制流问题：业务代码只声明自己需要某个效果，例如读取数据、记录日志、请求用户输入；外层处理器决定这个效果如何实现，并在处理后把结果送回原来的代码位置继续执行。

## 出现前提

- 程序既要保持业务代码清晰，又要处理异常、异步、日志、状态、输入输出等副作用
- 传统异常只能中断，不能在原位置恢复
- `async / await` 会把异步性质暴露给整条调用链，中间函数也被迫感知异步
- 纯函数语言中用 Monad 管理副作用较严谨，但写法和心智成本较高
- 复杂系统需要把「业务意图」和「运行时处理方式」分离

## 核心机制

### 1. `perform` 声明一个效果

`perform` 类似 `throw`，但它不是把错误抛出去后销毁调用栈，而是暂停当前位置，并把后续要执行的代码保存成一个 continuation。

```javascript
function getName(user) {
  let name = user.name;
  if (name === null) {
    name = perform 'ask_name';
  }
  return name;
}
```

这段代码表达的是：如果没有名字，就向外层声明「我需要一个名字」。它不关心名字来自弹窗、缓存、接口请求，还是测试桩。

### 2. 处理器接住效果

外层的 effect handler 接到这个请求后，决定如何处理。

```javascript
try {
  makeFriends(arya, gendry);
} handle (effect) {
  if (effect === 'ask_name') {
    resume with 'Arya Stark';
  }
}
```

### 3. `resume with` 回到原位置继续

`resume with` 会把值送回 `perform` 暂停的位置，让原函数继续执行。这里的关键不是「捕获」，而是「恢复」。

### 4. continuation 是被保存下来的后续流程

`perform` 暂停时，语言运行时会把「从这里往后还要做什么」保存下来。这个保存下来的后续流程就是 continuation。

原文中提到的 `resume with` 可以粗略理解为一个只能调用一次的回调；学术上更准确的说法是 one-shot delimited continuation。

## 收益

- 异常可以被就地修复后继续执行，而不是只能中断后善后
- 中间层函数不需要知道底层效果是同步还是异步，减少 `async` 传播
- 业务代码只写意图，具体处理方式由外层注入
- 日志、状态、依赖注入、测试替身、异步读取等关注点更容易分离
- 能解释 React Suspense、Hooks 等机制背后的设计思想

## 代价

- 控制流不再完全按函数调用顺序直线展开，调试难度会上升
- 如果语言和标准库没有整体支持，局部引入很容易变成难理解的魔法
- continuation 的保存、恢复、只调用一次等约束需要运行时严格管理
- 对 JavaScript 这类已经有异常、Promise、生成器和 async/await 的语言来说，完整引入可能过于强大

## 常见误区

- **「代数效应就是异常处理」**：它和异常相似，但核心差异是可以恢复原位置继续执行。
- **「它只是 async / await 的替代品」**：异步只是它能表达的一类效果，它也可以表达日志、状态、输入输出等副作用。
- **「React 已经实现了代数效应」**：React 没有在 JavaScript 语言层面实现代数效应，只是在 Suspense、Hooks 等机制中借用了相近思想。
- **「业务代码完全不用关心副作用」**：业务代码可以少关心处理细节，但仍然要清楚自己声明了什么效果，以及外层是否有对应处理器。

## 与 React 的关系

### Suspense

Suspense 的常见模型是：组件读取尚未就绪的数据时抛出 Promise，React 捕获后等待它完成，再重新渲染组件。

这不是严格意义上的代数效应，因为 JavaScript 的 `throw` 会丢掉原调用栈，React 通过重新渲染来近似「回到原处继续」。它相似的地方在于：组件只表达「我需要这个数据」，外层的 React 运行时决定等待、显示 fallback、恢复渲染。

### Hooks

`useState` 看起来像普通函数调用，却能知道自己属于哪个组件、对应哪一格状态。概念上可以把 React 内部的 dispatcher 理解为一种外层处理器：组件代码声明「我要状态」，React 运行时提供状态存取方式。

这也不是语言级代数效应，而是 React 在框架层面对「效果由外层运行时处理」这一思想的工程化实现。

## 适用场景

- 理解 Suspense 为什么能把「等待数据」交给 React 运行时处理
- 分析 `async / await` 为什么会沿调用链扩散
- 比较异常、Promise、生成器、continuation 这些控制流机制的边界
- 理解副作用管理为什么不只是「把函数写纯」，还包括「谁声明效果，谁解释效果」

## 一个真实例子

前端组件读取用户资料时，业务上只关心「拿到用户资料后渲染页面」。如果数据还没准备好，传统写法通常要把 loading、error、retry 分散进组件树或请求层。

Suspense 的思路更接近代数效应：组件声明自己需要数据，React 在外层接住未完成状态，先展示 fallback，等数据准备好后重新渲染组件。组件没有直接决定等待期间怎么显示，也没有把异步处理传染给每个中间组件。

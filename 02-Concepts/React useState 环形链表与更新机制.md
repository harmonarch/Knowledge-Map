# React useState 环形链表与更新机制

## 一句话定义

React useState 的环形链表与更新机制，是指 React 用一个循环更新队列暂存多次 state 更新，并按优先级分轮计算最终 state 的方式。

## 它解决什么问题

组件在一次交互里可能连续触发多次 `setState`，而 React 又不一定立刻执行渲染。它需要一种结构把更新按顺序暂存起来，支持批量消费，还要允许高优更新先响应、低优更新稍后补算。

## 出现前提

- 同一个 Hook 可能在一次事件里被连续多次更新
- 渲染不是每次 `setState` 后都同步发生
- React 需要把更新顺序保留下来，避免后一次更新覆盖前一次更新的语义
- React 18 之后调度系统引入了更明确的优先级区分，更新可能被跳过后延后重算

## 核心机制

1. 每个 `useState` Hook 都维护一个更新队列。
2. 每次调用 `setState`，React 会创建一个更新节点并挂到队列尾部。
3. 这个队列用环形链表表示，只保留 `queue.pending` 一个尾指针：
   - 当前尾节点是 `queue.pending`
   - 头节点是 `queue.pending.next`
4. 进入渲染时，React 从头到尾遍历这批更新，顺序计算最新 state。
5. 如果某些更新优先级不够，本轮先跳过；被跳过的位置以及其后的必要更新会进入 `baseQueue`，留到后续渲染重新计算。
6. 这样既能先把紧急交互反馈给用户，也能保证最终 state 仍然符合原始更新顺序。

## 收益

- 只用一个尾指针就能同时做到 O(1) 插入和头部访问，队列结构紧凑
- 多次 `setState` 可以集中处理，减少中间态渲染
- 高优更新可以先响应，交互延迟更低
- 被跳过的更新不会丢失，后续仍可按顺序得到正确结果

## 代价

- 更新流程比“来一次 setState 就立刻改一次值”更难理解
- 高低优更新混合时，用户短时间看到的 state 可能只是阶段性结果
- `baseQueue` 的补算规则提高了实现复杂度，也增加了调试门槛
- 只看最终结果时，容易忽略中间渲染轮次的存在

## 常见误区

- **“每次 setState 都会立刻改 state”**：更新通常先入队，再由后续渲染统一消费
- **“useState 只是保存一个值”**：它内部还维护更新队列、优先级和重算信息
- **“低优更新只是晚一点执行”**：一旦被跳过，后续相关更新也可能需要放进 `baseQueue` 重新计算
- **“startTransition 和 useDeferredValue 是同一个东西”**：前者控制更新产生时的优先级，后者控制某个值被消费时的延迟版本

## 一个真实例子

原始材料里的例子是一个混合优先级队列：

- 队列顺序：`[A:紧急, +1] → [B:低优, ×2] → [C:紧急, +3]`
- 初始 state = `0`
- 第一轮先处理高优更新：A 把 `0` 算成 `1`，跳过 B，再处理 C，把 `1` 算成 `4`
- 此时用户先看到 `4`
- 被跳过的 B 以及其后必须保序重算的 C 会进入 `baseQueue`
- 第二轮从 `baseState = 1` 重新算：B 把 `1` 算成 `2`，C 再把 `2` 算成 `5`
- 最终正确结果是 `5`

这个例子说明，React 的目标不是“每一轮都得到最终答案”，而是在保证最终顺序正确的前提下，优先让紧急交互尽快出现在界面上。

# Claude Code 省 Token 策略

> 来源：宝玉（@dotey）2026年4月13日 [推文](https://x.com/dotey/status/2043463546025992494)

## 问题

Claude Code 配额消耗过快（Max 用户一周额度两天耗尽），直觉操作（频繁 `/clear`、开新会话）反而更费 Token。

## 核心机制：提示缓存

- 大模型每次都要从头"读"完整输入，无记忆
- 提示缓存将之前的计算结果缓存，**读取缓存成本仅为全价的 1/10**
- 缓存只对"前缀"有效，有存活时间：主智能体 1 小时，子智能体 5 分钟
- **开新会话 = 缓存全部作废，触发全价重建**

## 三个反直觉策略

1. **缓存还热时，继续聊比开新会话便宜** — `/clear` = 反复为不变内容付全价写入费
2. **复杂任务一次做对比来回改三轮更省** — 每轮对话整个上下文都要重发
3. **长内容给路径，别往对话里贴** — 让 Claude 自己用工具检索，最便宜的 Token 是根本没进上下文的 Token

## 继续聊 vs 开新会话：决策表

| 条件 | 建议 |
|------|------|
| 任务没变，还在调同一个 bug / 写同一个模块 | 继续聊 |
| 距上一条消息不超过 1 小时，缓存还活着 | 继续聊 |
| 上下文内容对当前工作仍有用 | 继续聊 |
| 任务切换，老上下文对新任务无用 | 开新会话 |
| 闲置超过 1 小时，缓存大概率过期 | 开新会话 |
| 上下文被不相关内容塞满，噪音太多 | 开新会话 |

> 缓存还热、任务没换 → 继续聊。缓存过期、任务切换、噪音太多 → 果断重开。

## 1M 上下文窗口：慎用

- 1M 和 200K 同价，但缓存失效后全量重建代价极高
- 大多数日常会话在 80-120K 触发压缩，根本用不到 1M
- 上下文超过 200K 后模型表现明显下降，350K+ 基本靠运气

建议配置自动压缩阈值：`CLAUDE_CODE_AUTO_COMPACT_WINDOW=200000`

如需禁用 1M 上下文：`CLAUDE_CODE_DISABLE_1M_CONTEXT=1`

## 六条操作规则

1. **用 Sonnet 做日常工作** — Opus Token 消耗约为 Sonnet 两倍，大多数编码任务 Sonnet 足够
2. **别在会话中间换模型** — 提示缓存按模型隔离，切模型 = 缓存从零开始
3. **精简 CLAUDE.md，控制技能数量** — 控制在 200 行以内，没在用的 MCP 服务记得关掉
4. **命令行优先，MCP 其次** — `gh` 比 GitHub MCP 消耗 Token 少得多
5. **先花一点 Token 做计划** — 复杂任务先进计划模式，总成本往往更低
6. **用 permissions.deny 限制阅读范围** — 排除 node_modules、构建产物等无关文件

## 委派工作

- **子智能体**：独立上下文，完成后只返回简短摘要，隔离详细输出不污染主会话
- **智能体团队**：每个成员独立实例，计划模式下消耗约为标准会话 7 倍，慎用
- **Codex 插件**：`claude mcp add codex -- npx -y @openai/codex-plugin-cc`，适合结构化 bug 修复、代码审查、写测试

## 代价

- 继续聊意味着上下文逐渐膨胀，需要主动管理噪音
- 省缓存策略需要用户有意识地判断"任务是否切换"

## 边界

- 不适用于非 Claude Code 的 LLM 客户端（缓存机制不同）
- 缓存存活时间（1h/5min）为当前值，可能随 Anthropic 更新变化

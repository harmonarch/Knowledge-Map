# React 组件拆分与 Controller 粒度

## 一句话定义

React 组件拆分与 Controller 粒度，是用语义、交互周期和数据边界来决定组件怎么拆、数据逻辑放哪一层的方法。

## 它解决什么问题

复杂页面里，最容易失控的不是“组件不够多”，而是拆分标准不清楚：

- 看到代码变长就拆，但拆出来的部分没有清楚职责
- 为了复用而提前抽象，结果公共组件变得难改
- 一个页面里多个接口、多个 Loading、多个数据转换混在一起
- controller 放在页面级，最后变成所有数据逻辑的大容器
- 上层组件既关心布局，又关心接口细节，还要处理数据合并

这个概念要解决的是：如何用稳定的判断标准，把页面复杂度切成可以独立理解的几层。

## 出现前提

- React 页面已经不只是简单展示，而是包含多个区域、多个异步状态和多种交互
- 一个组件的数据可能来自多个 API，需要先合并再展示
- 页面代码变长后，维护者需要在阅读时“折叠”掉暂时不关心的细节
- 团队需要判断某段逻辑应该留在组件内、抽到 controller，还是进入 service
- 组件边界和 Loading 边界开始影响用户体验，而不只是影响代码组织

## 核心机制

### 1. 先总后分，再回到整体

组件拆分不应该从局部代码片段开始，而应该先看完整页面：

- 页面分成哪些语义区域
- 用户会触发哪些交互
- 哪些数据需要一起出现、一起更新
- 哪些部分可以被当成独立黑盒理解

先理解整体，再拆出局部，最后再确认这些局部能不能组合回清楚的页面结构。

### 2. 拆分的目的主要是降低阅读负担

组件拆分不等于复用。很多组件只服务当前页面，也仍然值得拆出来。

更实用的判断标准是：这段逻辑后续阅读时能不能被“折叠”起来。比如评论列表、商品信息、推荐区都可以各自封装成有名字的区域。维护商品信息时，不需要同时读懂评论和推荐。

代码行数只能作为提醒，不是硬规则。一个 200 行左右的组件可能已经值得拆，也可能因为逻辑集中而暂时不拆。关键是拆出来以后是否真的让问题更小。

### 3. 语义决定边界

每个组件、函数、controller、service 都要能说清楚“它负责什么”。

如果拆出来的部分只能叫 `PartA`、`handleData`、`commonLogic`，说明语义还没有想清楚。没有语义的拆分只会增加跳转，不会降低复杂度。

语义清楚之后，边界会自然出现：

| 层级 | 语义 |
|---|---|
| 组件 | 展示一个完整的界面区域 |
| controller | 产出这个区域可以直接使用的数据 |
| service | 请求和合并更底层的原始数据 |

### 4. Loading 边界由交互设计决定

Loading 的数量不应该由 API 数量决定，而应该由用户看到的交互周期决定。

一个组件即使依赖三个接口，只要它们在界面上表现为同一段加载和更新过程，就应该对应一个 Loading 和一个组件边界。接口差异应该由 service 或 controller 抹平，不应该泄漏到 View。

反过来，如果页面里两个区域可以独立更新、独立失败、独立展示 Loading，它们就更适合拆成不同组件和不同 controller。

### 5. 每次只想一层

面对复杂页面时，不要同时思考组件结构、接口细节和异常处理。更好的方式是分层推进：

1. 组件层只判断页面分成哪几块，以及每块需要什么数据
2. controller 层只判断怎么产出组件需要的数据形状
3. service 层只判断怎么请求接口、并发合并和处理底层异常

上层把下层当黑盒，才能避免一个人同时背负太多问题维度。

### 6. Controller 跟着组件走

controller 不应该默认做成页面级的大统一入口。更稳定的判断规则是：

```text
controller 的粒度跟着组件走
组件的粒度跟着 Loading 走
Loading 的粒度跟着交互设计走
```

这意味着，一个页面可以有多个 controller。每个 controller 服务一个语义清楚、更新周期清楚的组件区域，让组件只面对稳定的数据结果。

## 收益

- 组件拆分有明确标准，不再只靠代码行数或个人习惯
- 页面可以按语义区域阅读，维护时更容易忽略无关细节
- Loading 状态更贴近真实交互，不会被后端接口数量牵着走
- controller 不容易膨胀成页面级大容器
- service、controller、View 的边界更清楚，复杂度可以分层处理

## 代价

- 需要先理解页面整体，不能看到局部代码就马上拆
- 语义命名要求更高，命名不清楚时拆分会变成负担
- controller 数量会增加，需要稳定的目录约定
- Loading 边界依赖交互设计，前期需要和设计、产品理解一致
- 简单页面如果强行套完整分层，会增加不必要的文件和跳转

## 常见误区

- **“拆组件是为了复用”**：复用只是结果之一，更常见的目的其实是降低阅读和维护成本。
- **“接口有几个，Loading 就有几个”**：用户感知的是交互周期，不是后端接口数量。
- **“controller 应该统一放在页面级”**：页面级 controller 容易变成杂糅容器，粒度更应该跟着组件和 Loading 走。
- **“代码长就一定要拆”**：拆分要看能否形成语义边界，而不是只看行数。
- **“解耦就是把代码分文件”**：真正的解耦是每一层只处理自己的问题，不越级关心下一层细节。

## 适用场景

- 一个 React 页面有多个独立区域，且每个区域都有自己的加载和更新过程
- 页面需要合并多个接口，但组件不应该理解接口细节
- `index.tsx` 同时承担布局、请求、数据转换和异常处理，已经难以阅读
- controller 逐渐膨胀，需要重新划分粒度
- 团队对“什么时候拆组件、什么时候抽 controller”没有共同标准

## 一个真实例子

商品详情页可以拆成 `ProductInfo`、`CommentList`、`Recommendations` 三个区域。页面入口只负责布局和组合，不直接关心每个区域的数据怎么来。

`ProductInfo` 可能需要商品基础信息、库存、促销价三个接口。如果用户看到的是一个整体加载过程，就让 `ProductInfo` 对应一个 Loading；内部由 service 并发请求并合并原始数据，再由 controller 转成组件需要的结构。

评论区和推荐区如果能独立加载、独立失败、独立刷新，就分别拥有自己的组件和 controller。这样页面不会出现一个大 controller 同时处理商品、评论和推荐，也不会让 View 直接面对多个接口的细节。

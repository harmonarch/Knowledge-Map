# DDD 领域驱动设计

## 一句话定义

DDD 领域驱动设计，是用业务领域而不是技术层级作为代码组织中心，让同一块业务的接口、逻辑和数据访问尽量内聚在一起的架构思维。

## 它解决什么问题

传统 MVC 会把代码按技术职责分成 controller、service、repository 等层。这个方式在项目早期很清楚，但业务变多后会出现几个问题：

- 同一个业务的代码散落在多个大目录里
- 修改一个业务时，需要在 controller、service、repository 之间来回找文件
- 某一层目录变成几百上千个文件，难以定位
- 业务之间互相调用，拆分服务时牵一发动全身
- 团队表面上分了层，实际没有清楚的业务边界

DDD 要解决的不是“是否分层”，而是“业务边界是否成为架构里的第一等公民”。

## 出现前提

- 后端已经不只是少量接口，而是包含多个稳定业务模块
- 按 controller、service、repository 分层后，业务代码仍然分散
- 一个业务改动经常需要跨多个技术目录搜索
- 团队开始考虑微服务、模块拆分或独立交付
- 业务规则比简单增删改查更重要，需要被明确表达和保护

## 核心机制

### 1. MVC 是按技术职责分层

MVC 的基本分工是：

| 层 | 职责 |
|---|---|
| Controller | 接收请求、校验参数、调用业务逻辑、返回响应 |
| Service | 处理业务规则、组合数据、完成业务流程 |
| Repository / DAO | 读写数据库或其他持久化资源 |
| View | 返回给调用方的展示结果，前后端分离后通常是 JSON |

这套分层的价值是让请求处理、业务规则和数据访问各自有位置。问题是，它只解决技术职责的划分，不解决业务边界的划分。

### 2. 反转目录主轴

传统目录往往是技术层级在上：

```text
controller/
  users/
  orders/
service/
  users/
  orders/
repository/
  users/
  orders/
```

DDD 更强调业务领域在上：

```text
users/
  controller/
  service/
  repository/
orders/
  controller/
  service/
  repository/
```

差别不只是目录写法，而是谁拥有优先级。前者优先回答“这段代码是什么技术职责”，后者优先回答“这段代码属于哪个业务”。

### 3. 领域模块应该自包含

一个领域模块内部可以继续分 controller、service、repository，但这些层首先服务于同一个业务边界。

这样做的目标是：修改用户业务时，主要进入 `users` 目录；修改订单业务时，主要进入 `orders` 目录。技术分层仍然存在，但它不应该压过业务边界。

### 4. 单向依赖保护分层

领域内部仍然需要保持单向依赖：

```text
Controller -> Service -> Repository / DAO -> 基础设施
```

上层可以依赖下层，下层不应该反向依赖上层。比如 Repository 不应该调用 Service，Service 不应该调用 Controller。

这条规则的意义是让每层可以被替换。REST Controller 改成 GraphQL 入口时，Service 不应该跟着重写；数据库访问方式变化时，Controller 不应该感知。

### 5. DDD 指导微服务拆分

DDD 是设计思维，微服务是部署形态。更合理的顺序是先识别领域边界，再决定哪些领域值得拆成独立服务。

如果没有领域边界，直接按接口数量或团队喜好拆微服务，很容易得到一堆互相强依赖的小服务。服务数量增加了，业务内聚没有变好，沟通和排障成本反而上升。

### 6. 和前端组件化是同一种内聚思想

后端 DDD 和前端组件化面对的场景不同，但底层问题相似：不要只按技术类型归类，而要让经常一起变化的东西住在一起。

| 场景 | 内聚单位 | 典型内容 |
|---|---|---|
| 后端 DDD | 业务领域 | controller、service、repository、领域规则 |
| 前端组件化 | UI 功能单元 | 视图、交互逻辑、样式、局部数据处理 |

共同目标是：改一个业务或功能时，尽量只动一个清楚的边界。

## 收益

- 同一业务的代码更集中，定位和修改成本更低
- 技术分层仍然保留，但不再压过业务边界
- 业务模块之间边界更清楚，后续拆服务更容易
- 领域规则更容易被看见，而不是散落在各层工具代码里
- 团队可以围绕业务模块协作，而不是围绕 controller、service 这些技术目录协作

## 代价

- 需要先识别业务边界，不能只按接口或数据库表机械拆分
- 小项目过早使用完整 DDD 结构，会增加目录和理解成本
- 领域之间的共享逻辑需要谨慎处理，否则容易形成新的公共杂物层
- 如果团队没有共同的领域语言，目录名只是换了一层包装
- 微服务拆分如果只学形式，不看业务内聚，会把复杂度转移到服务通信上

## 常见误区

- **“DDD 就是把目录改成业务名”**：目录只是结果，真正重点是业务边界和规则内聚。
- **“DDD 不需要 MVC 分层”**：DDD 不反对分层，它反对技术分层压过业务边界。
- **“用了微服务就是 DDD”**：微服务是部署方式，DDD 是识别和保护业务边界的方法。
- **“所有项目都应该上 DDD”**：简单 CRUD 或业务很少的项目，完整 DDD 结构可能比问题本身更重。
- **“Repository 只要能查数据库就行”**：Repository 应该服务领域边界，不应该把业务判断散落到数据库访问细节里。

## 适用场景

- 后端业务模块已经变多，按技术目录查找代码开始困难
- 一个业务规则散落在多个 service、controller 或 repository 中
- 团队准备拆分模块、微服务或独立交付边界
- 需要把业务语言、代码结构和团队协作边界对齐
- 想判断前端组件化、BFF、后端领域模块之间的内聚逻辑是否一致

## 不适用场景

- 只有少量接口和简单 CRUD，业务规则还不复杂
- 团队还没有稳定业务边界，频繁重命名领域会制造额外成本
- 只是想通过换目录结构解决所有架构问题
- 没有能力维护领域之间的依赖规则和共享逻辑边界

## 一个真实例子

一个电商系统最初按 `controller/`、`service/`、`repository/` 分目录。用户、订单、营销的代码分别散在三层里。后来要修改订单取消规则时，需要同时找订单 controller、订单 service、库存 service、优惠券 service 和多个 repository，修改路径很长。

更好的做法是先识别订单领域，把订单相关入口、业务流程、数据访问和领域规则收拢到 `orders/` 模块中。库存和优惠券如果属于其他领域，就通过清楚的接口协作，而不是让订单代码直接深入修改它们的内部细节。

这样未来如果订单模块需要独立部署，至少已有一个比较完整的业务边界；即使暂时不拆微服务，日常维护也更容易定位。

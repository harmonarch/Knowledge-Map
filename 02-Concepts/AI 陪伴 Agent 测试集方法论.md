# AI 陪伴 Agent 测试集方法论

## 一句话定义

AI 陪伴 Agent 测试集方法论，是把「陪伴质量」拆成可复用测试用例、反面约束、人工评审、自动评分和上线门禁的一套验证方法。

## 它解决什么问题

AI 陪伴 Agent 的质量很难只靠单元测试判断。它不像普通函数一样有唯一正确答案，同一句用户输入在不同记忆、情绪和上下文里，合理回复也不同。

真正要防的是三类问题：

- 记忆和上下文用错：把压力驱动误判成兴趣偏好，或强行关联无关记忆
- 陪伴语气失真：说教、过度建议、暴露 AI 身份、模板化安慰
- 版本迭代退步：新 Prompt 平均分变高，但关键场景变差

所以测试集不是为了让 Agent 逐字匹配标准答案，而是为了约束它「应该表现出什么意图」和「绝对不能踩哪些线」。

## 出现前提

- Agent 已经有基础对话能力，但回复质量不稳定
- 系统里存在用户画像、长期记忆、近期上下文、情绪判断等变量
- Prompt 或管线会持续迭代，需要防止旧能力退步
- 团队已经无法只靠少量人工体验判断版本质量

## 核心机制

### 1. 先写 10 条骨架用例

不要一开始追求大规模测试集。先围绕最重要的能力各写 2-3 条：

- 倾听能力
- 记忆准确性
- 推理质量
- 情绪判定
- 禁忌合规
- 连续输入处理

每条用例回答三个问题：

| 字段 | 作用 |
|---|---|
| setup | 给 Agent 什么环境，包括用户画像、记忆、近期上下文 |
| input | 用户这次说了什么 |
| expected | Agent 应该怎么做，以及不能怎么做 |

其中最重要的是 expected。对陪伴 Agent 来说，`should_not_contain` 往往比 `should_contain_intent` 更有价值。因为回复不需要逐字一致，但必须避免错误方向。

### 2. 从正面用例扩展到反面用例

测试集不能只写「应该做什么」，必须覆盖「不应该做什么」：

| 场景 | 约束 |
|---|---|
| 用户说「烦死了这破天气」 | 不应判为 vulnerable |
| 用户说「今天吃了个汉堡」 | 不应强行关联情绪 |
| 任意陪伴场景 | 不应说「作为一个 AI」 |
| 用户处于脆弱状态 | 不应急着给建议 |
| crisis 场景 | 不应说空泛的「会好的」 |

反面用例的价值在于防退步。Agent 可以有不同表达，但不能越过边界。

### 3. expected 写意图，不写标准答案

陪伴 Agent 的 expected 不应该保存一条理想回复原文，而应该保存这条回复背后的特征：

- 应该包含的意图
- 不应该出现的词或行为
- 语气要求
- 最大句数
- 推理必须成立和不得成立的判断

例如理想回复是「每天都喝呀，找到喜欢的店了吗？对了新工作那边怎么样？」时，expected 应写成：

- `should_contain_intent`: 自然地从咖啡过渡到关心新工作
- `should_not_contain`: 压力、你应该、注意身体
- `tone`: 轻松关心
- `max_sentences`: 3

这样测试的是质量边界，而不是文案复刻。

### 4. 按阶段使用测试集

| 阶段 | 用例数 | 方式 | 重点 |
|---|---:|---|---|
| Prompt 初稿 | 10-20 条 | 手动跑 | 快速确认核心方向 |
| 管线搭建 | 50-100 条 | 脚本输出 + 人工审查 | 看完整流程是否稳定 |
| 自动评测 | 50 条以上 | 评审 LLM 打分 | 扩大覆盖面，减少人工负担 |
| 上线前 | 100-200 条 | 全自动 + 对比报告 | 防止版本退步 |

早期不必急着做自动评分。先把最关键的 10-20 条跑顺，确认 Prompt 方向对，再扩大规模。

### 5. 评审 LLM 必须严格

当用例超过人工可审范围时，可以用独立 LLM 做评审。但评审 Prompt 必须明确要求：

- 3 分只是刚好及格
- 4 分代表不错
- 5 分代表无可挑剔
- justification 必须引用 Agent 回复中的具体原文

如果不这样约束，LLM 评审会倾向给高分，导致测试集失去筛选能力。

### 6. 用对比报告看退步

每次改 Prompt，不只看整体得分，还要看退步用例：

- 整体得分是否上升
- 禁忌 pass 率是否下降
- 推理准确性是否变好
- 回复自然度是否变差
- 锚定用例是否波动过大

平均分变高不代表版本更好。如果进步 3 条、退步 5 条，说明改动可能只是优化了容易场景，牺牲了关键边界。

### 7. 上线前设置 Gate

Prompt 发版前需要质量门禁，例如：

- 整体得分达到及格线
- 禁忌 pass 率达到 95% 以上
- 情绪判定均分达到 4.0 以上
- 新增退步用例不超过 3 条
- 锚定用例分数变化不超过 0.5

Gate 的意义不是追求完美，而是防止明显退步版本进入生产环境。

### 8. 上线后持续回捞

测试集不是一次性资产。上线后要从生产日志里回捞新问题：

1. 采集用户说「你不懂」、快速结束对话等负向信号
2. 自动标记为待审对话
3. 每周人工审核一批
4. 发现新问题后写成测试用例
5. 加入测试集并重新评测当前版本

陪伴 Agent 的失败模式会随着真实用户变化。测试集必须持续吸收新失败，否则会越来越像旧版本的自我安慰。

## 收益

- 把主观的陪伴质量变成可以反复检查的测试资产
- 让 Prompt 修改有回归依据，不靠体感判断
- 通过反面用例减少说教、误判、过度推理和模板化安慰
- 让上线前质量门禁有明确标准
- 让线上失败能反哺测试集，形成持续改进闭环

## 代价

- expected 需要人工提炼，不能直接复制理想回复
- 自动评审需要额外设计 Prompt，否则评分会虚高
- 测试集需要长期维护，过旧的用例会降低判断价值
- 陪伴质量仍然无法完全自动化验证，关键用例需要人工抽查

## 常见误区

- 只写正面用例，不写反面用例
- 把 ideal response 当成 expected，要求 Agent 逐字模仿
- 只看平均分，不看退步用例
- 过早追求全自动评分，忽略早期人工审查
- 把测试集当静态文件，不从线上失败里持续补充
- 用同一个 Agent 自己生成、自己评审，缺少独立约束

## 适用场景

- AI 伴侣、情感陪伴、长期个人助手等对语气和记忆敏感的 Agent
- Prompt、记忆检索、情绪状态机持续迭代的系统
- 需要上线前质量门禁和版本对比报告的团队
- 想把真实用户负反馈沉淀成测试资产的产品

## 一个真实例子

用户画像显示「小林是前端工程师，两周前刚入职字节跳动」，近期上下文里用户说「今天又加班到九点」。这时用户输入「最近每天都要喝咖啡才行」。

好的 expected 不应该要求 Agent 说出某一句固定话，而应该约束：

- 必须能推断咖啡可能和新工作适应压力有关
- 不能直接断定用户单纯喜欢咖啡
- 回复要自然关心新工作适应
- 不要直接说「你是不是压力大」
- 不要说「你应该注意身体」
- 最多 3 句话

这个例子说明：陪伴 Agent 测试的重点不是答案唯一，而是边界清楚。

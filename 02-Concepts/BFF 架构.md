# BFF 架构

## 一句话定义

BFF（Backend For Frontend）是在前端和后端之间增加一层面向界面需求的数据适配层，用来聚合接口、转换字段、裁剪数据，让前端拿到可以直接使用的数据形状。

## 它解决什么问题

后端接口通常按领域模型、数据库结构或微服务边界设计，前端页面却按展示场景取数。两者的数据形状经常不一致：

- 一个页面需要多个后端接口的数据
- 后端返回的是通用领域数据，前端还要补字段、改字段名、转树形结构
- 不同端对同一业务数据需要不同字段
- JSX 中出现大量防御性判断、字段拼接和数据转换

BFF 要解决的不是“前端怎么请求接口”，而是“前端应该面对一个什么样的数据源”。目标是让 View 层的数据来源唯一、结构稳定、拿来即用。

## 出现前提

- MVC 中的 Model 层开始变重，既要请求接口，又要校验数据，还要做格式转换
- 后端接口无法完全按页面需求定制，或者多个后端服务需要聚合
- 前端页面需要稳定的数据形状，不希望每个组件都理解后端返回细节
- 多端产品共用后端，但 Web、App、小程序等端的数据需求不同
- 前后端团队协作成本高，需要一个更可控的适配边界

## 核心机制

### 1. 前端项目内也可以使用 BFF 思想

BFF 不一定等于单独起一个 Node 服务。小到中型 React 项目里，可以直接用文件分层表达 BFF 思想：

| 层 | 文件 | 职责 |
|---|---|---|
| API 层 | `api.ts` | 只负责调用后端接口，尽量保持后端原始数据格式 |
| Service 层 | `service.ts` | 聚合多个接口，处理并发请求，形成页面所需的单一数据源 |
| Controller 层 | `controller.ts` | 校验数据、补字段、转换结构，输出 View 可直接消费的数据 |

这套分层的核心是让 `index.tsx` 少做数据处理。View 层只负责展示，不负责猜测接口字段是否存在，也不负责把多个接口拼成页面模型。

### 2. Node BFF 是独立中间层

当需求超过前端项目内部分层能承担的范围时，可以引入独立 Node BFF 服务。它位于浏览器和后端服务之间，负责：

- 在服务端聚合多个后端接口
- 按不同端裁剪字段
- 做敏感数据脱敏
- 加缓存、限流、熔断
- 在 SSR / SEO 场景下提前准备页面数据
- 屏蔽微服务接口的复杂性，让前端只请求一次

独立 Node BFF 的代价是它变成了一个需要部署、监控和维护的服务，不能只把它当成前端项目里的小工具。

### 3. 职责边界按数据形状判断

一个实用判断规则是：谁定义数据形状，谁负责相应变换。

| 职责 | 归属 | 判断标准 |
|---|---|---|
| 提供完整、自洽的领域数据 | 后端 | 这个数据在业务上是否是一个整体 |
| 转成某个端或某个页面需要的形状 | 前端 / BFF | 这个变换是否只服务展示需求 |
| 保证一致性、事务、权限和审计 | 后端 | 这个逻辑是否影响业务正确性 |
| 字段重命名、排序、轻量格式化 | 前端 / BFF | 这个逻辑是否只影响界面呈现 |

### 4. 用接口契约减少推诿

BFF 容易制造灰色地带：同一段逻辑前端能做、后端也能做。要避免互相推诿，需要把口头约定变成契约：

- 用 OpenAPI / Swagger 描述接口结构
- 用 schema 校验返回数据
- 用契约测试检查接口变更
- 明确 BFF 层归属，一般应由前端团队负责

### 5. BFF 直连数据库是危险信号

如果 Node BFF 绕过后端 API 直接读写数据库，它就不再只是中间层，而是变成了一个没有被当成后端管理的后端。

这里的红线是：写操作必须走后端 API。BFF 如果因为历史原因必须读数据库，也应使用 ORM 或 Repository 抽象，并把它记录为技术债务。

## 收益

- View 层更干净，页面代码只面对稳定、可直接使用的数据
- 多接口聚合从组件中移出，数据来源更集中
- 不同端可以拿到各自需要的数据形状，减少前端重复转换
- 微服务后端的复杂性可以被中间层屏蔽
- 在 SSR、缓存、限流、脱敏等场景中，可以把数据处理前移到服务端

## 代价

- 独立 Node BFF 会增加部署、监控、鉴权、缓存和故障排查成本
- 职责边界不清时，容易变成前后端互相推诿的新战场
- BFF 层过厚会复制后端业务逻辑，长期形成第二套后端
- 直连数据库会绕过后端校验、权限和审计，带来数据一致性与安全风险
- 只为简单单体后端和单一前端引入独立 BFF，通常得不偿失

## 常见误区

- **「BFF 必须是一个 Node 服务」**：BFF 首先是一种数据适配思想，前端项目内部的 `api.ts`、`service.ts`、`controller.ts` 分层也可以承载它。
- **「BFF 可以替代后端」**：BFF 服务前端体验，不应该接管领域规则、事务一致性和核心权限。
- **「所有接口聚合都应该放 BFF」**：如果只是一个页面内部的小转换，前端内部 controller 就够了。
- **「Node BFF 直连数据库更快」**：短期可能少写接口，长期会让权限、审计、表结构变更和性能责任全部失控。
- **「BFF 越厚越灵活」**：BFF 过厚会复制后端逻辑，最终增加系统复杂度。

## 适用场景

- 一个页面需要聚合多个后端接口，且前端希望只面对一个数据源
- Web、App、小程序等多个端共享后端，但展示字段和数据形状不同
- 后端是微服务架构，前端不适合直接理解多个服务的调用关系
- 需要在服务端做字段裁剪、脱敏、缓存、限流或熔断
- Next.js 等 SSR 场景中，页面数据本来就需要在服务端准备
- 前后端团队接口协调成本高，前端需要可控的适配层

## 不适用场景

- 单体后端、单一前端、接口结构简单，前后端沟通顺畅
- 只是少量字段重命名或展示格式转换
- 团队没有能力维护额外服务的部署、监控和安全边界
- BFF 的真实用途是绕过后端接口直接操作数据库

## 一个真实例子

一个订单详情页需要订单基础信息、物流轨迹、优惠券、售后状态和用户权限。后端按领域拆成多个接口返回，每个接口的数据都比较完整，但页面只需要其中一部分字段，还要把它们合成一个展示模型。

如果直接在 JSX 里分别请求和拼接，页面会充满 loading 判断、字段判空和格式转换。更好的做法是：`api.ts` 对应原始接口，`service.ts` 并发请求并合并数据，`controller.ts` 校验和转换成 `OrderDetailViewModel`，View 只负责渲染这个模型。

如果同样的数据还要服务 Web、App 和小程序，并且需要服务端缓存、脱敏和权限裁剪，就可以考虑独立 Node BFF；如果只是这个页面内部使用，前端项目内分层就足够。

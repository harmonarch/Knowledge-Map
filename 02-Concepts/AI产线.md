# AI产线

## 一句话定义

AI产线是一条**端到端的、以AI为执行主体的软件生产流水线**——自然语言需求进去，可工作的软件出来。它不是CI/CD管线的升级版，而是一个以概率性智能体为执行主体、需要维护"智能体的认知行为"而非"脚本和配置"的新型工程系统。

## 它解决什么问题

当前Copilot模式下，AI只是人的辅助工具，人仍然是整条链路中唯一的认知主体和质量防线。AI产线将AI从"辅助工具"提升为"执行主体"，人可以退到设计/维护/偏差仲裁位，真正实现"以AI为中心的工程化软件生产"。

## 出现前提

- 大模型具备了执行软件工程各环节任务的能力
- 每个节点都有对应的确定性裁判机制
- 团队有能力设计和管理概率性系统的工程行为

## 核心机制

**产线vs CI/CD的本质区别**：

| 维度 | 今天的CI/CD管线 | AI产线 |
|---|---|---|
| 执行主体 | 确定性脚本 | 概率性智能体 |
| 输入 | 人写好的代码 | 自然语言需求 |
| 管线内的"决策" | 几乎没有 | 大量 |
| 失败模式 | 确定性的 | 概率性的 |
| 需要维护的是 | 脚本和配置 | 智能体的认知行为 |

**五类产线岗位**：

| 岗位 | 职能 | 类比经典工程 |
|---|---|---|
| AI产线架构师 | 设计软件生产流程 | 工艺工程师（大野耐一设计丰田生产系统） |
| 认知SOP工程师 | 把领域知识沉淀为AI可执行的流程模板 | 工艺标准编写员 |
| 偏差检测工程师 | 设计自动验证/反例生成/输出审查机制 | 芯片良率工程师 |
| AI产线调优师 | 持续优化提示词、知识库、模型选型 | 台积电制程工程师 |
| 认知边界守卫 | 处理AI拉不回的偏差 | 核电站高级值班工程师 |

**AI产线架构师**设计的是"生产软件的系统的结构"——哪些环节由哪种AI负责、验证环节放在哪里、人在什么节点介入。最接近"技术VP+首席架构师"的结合体。

**分治单元的切分**必须对齐"确定性裁判"的边界：切出来的每个单元，输出能不能被独立验证？能独立验证 = 有确定性裁判 = AI可以形成闭环。

**正确的切分方式**：按"端到端业务价值"切（如商品域、交易域、库存域、用户域），而不是按"技术模块"切（如前端团队、后端API团队、数据库团队）。

## 收益

- 实现端到端的AI主导软件生产流程
- 团队积累自己的工程资产（而非为AI厂商打工）
- 方法论可指数加速复制到其他产线

## 代价

- 产线建设初期投入大，需要艰难探索
- 需要全新的岗位和技能组合
- 调优变量空间极大且互相耦合，反馈信号滞后且模糊

## 常见误区

- 把AI产线理解为"CI/CD管线+AI插件"——两者的执行主体、失败模式、维护对象完全不同
- 逐节点替换——每个AI节点被人节点切成孤岛，无法形成端到端的AI主导流程
- 按技术模块切分治单元——应该按端到端业务价值切

## 适用场景

- 编码+测试产线（最适合先建）
- 发布运维产线
- 系统设计产线（中期目标）
- 需求分析产线（远期目标）

## 一个真实例子

第一条产线（编码产线）：AI接收自然语言需求 → AI生成代码 → 编译器+单测验证 → AI修复失败测试 → 再次验证 → 直到全绿。人在这个产线中的角色不是写代码或review代码，而是：设计产线拓扑、维护验证规则、处理AI拉不回的偏差、持续优化prompt和知识库。

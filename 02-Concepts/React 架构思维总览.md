# React 架构思维总览

## 一句话定义

React 架构思维是一套用目录结构、组件边界和状态流向来降低前端项目维护成本的方法，核心是把数据逻辑、界面展示和状态变化分清楚。

## 它解决什么问题

前端项目变复杂后，真正难的不是写出第一个页面，而是长期修改时还能知道：

- 某段代码应该放在哪里
- 一个组件为什么重新渲染
- 一次状态变化是谁触发的
- 页面逻辑、接口逻辑和展示逻辑是否互相缠在一起

如果没有结构约束，项目很容易变成页面、组件、接口、状态和工具函数混放。短期看写得快，后期每次改动都要在多个文件之间猜关系。

## 出现前提

- React 项目不再只是少量展示组件，而是包含页面、接口、状态、表单和复杂交互
- 多个页面开始共享组件、接口或数据处理逻辑
- 同一个状态可能被多个地方读取和修改
- 团队协作时，不同人需要并行修改页面、数据和组件
- 项目需要在出问题时追踪「数据从哪来、在哪里变、为什么影响 UI」

## 核心机制

### 1. 目录先按语义分层

常见顶层目录可以按职责划分：

- `pages`：页面入口
- `components`：跨页面共享组件
- `utils`：通用工具函数
- `hooks`：可复用的状态和副作用逻辑
- `api`：跨页面共享接口

这个结构的意义不是追求目录好看，而是让新代码有默认归属，减少每次新增文件时的判断成本。

### 2. 页面内部优先内聚

与某个页面强相关的代码，先放在这个页面目录内，例如：

- `index.tsx`：页面展示
- `api.ts`：页面专用接口
- `model.ts`：页面专用数据处理
- `index.css`：页面样式
- `index.d.ts`：页面类型

判断规则是：只在这个页面内部复用，就留在页面目录；被多个页面稳定使用，再提升到全局公共目录。

这条规则可以避免过早抽象。很多“以后可能复用”的代码，最后并不会真的复用；提前提升只会让全局目录变脏。

### 3. 用 MVC 思维拆开关注点

MVC 在 React 中不是照搬传统框架，而是一种拆分问题的方式：

| 层 | 关注点 | React 中的常见位置 |
|---|---|---|
| Model | 数据从哪来、长什么样、如何转换 | `api.ts`、`model.ts`、store、数据处理函数 |
| View | 用户看到什么 | JSX、展示组件、样式 |
| Controller | 什么时候更新、由什么事件触发 | 自定义 hook、事件处理、状态开关 |

React 组件天然会同时包含 View 和 Controller，所以需要有意识地把复杂逻辑抽到 hook 或 model 中，而不是让 `index.tsx` 同时承担所有职责。

### 4. Redux 解决的是状态修改可追溯

Redux 不是 MVC 里的某一个层，而是横跨 Model 和 Controller 的状态管理方案。

| Redux 概念 | 更接近的职责 | 原因 |
|---|---|---|
| Store | Model | 保存应用状态 |
| Reducer | Model | 定义状态如何变化 |
| Action / dispatch | Controller | 触发一次状态变化 |

它解决的重点不是「怎么取数据」，而是「谁能改数据、怎么留下记录」。

Redux 把状态变化压成单向流程：

```text
View -> dispatch(action) -> Reducer -> 新 State -> View 更新
```

这样做的收益是状态变化可追踪。出了问题时，可以沿着 action 和 reducer 回看每一次变化，而不是在多个组件里猜是谁改了状态。

## 收益

- 新代码更容易找到合适位置，项目结构不容易失控
- 页面专属逻辑保持内聚，跨页面逻辑再提升复用
- 数据逻辑和展示逻辑分开后，改接口不必牵动 UI，改 UI 也不必重写数据处理
- 状态变化路径更清楚，排查问题时更容易追溯来源
- 团队协作时，页面、组件、接口、状态的修改边界更明确

## 代价

- 简单页面硬套分层会增加文件数量和理解成本
- Controller 在 React 中边界不天然清晰，需要团队形成一致习惯
- 过早把页面内部逻辑提升到全局目录，会让公共层变成杂物间
- Redux 这类集中式状态管理会带来额外模板和约束，小项目可能不划算
- 分层越多，数据流跨层追踪越依赖命名、目录和团队纪律

## 常见误区

- **「所有组件都要拆成 MVC」**：纯展示组件和简单表单不需要强行拆层，拆分应该服务于复杂度。
- **「能复用就应该立刻放全局」**：真正稳定跨页面复用后再提升，页面内复用先保持局部内聚。
- **「Redux 是 MVC 的 Model 层」**：Redux 同时承担状态保存、状态计算和状态变更触发，不只是 Model。
- **「架构就是选工具」**：Redux、zustand、目录规范都是手段，真正目标是关注点分离、可维护和可追溯。
- **「文件越拆越专业」**：拆分如果不能降低理解成本，只是在制造更多跳转。

## 适用场景

- React 项目从单页面演进到多页面、多状态、多接口
- 页面内部逻辑开始变长，需要拆分数据、展示和更新触发
- 多个页面开始共享组件或接口，但还不确定抽象边界
- 状态问题难排查，需要明确「谁改了状态」
- 团队协作时需要统一文件放置和模块边界

## 一个真实例子

一个用户管理页面最开始只有列表展示，所有逻辑写在 `index.tsx` 里没有问题。后来它增加了搜索、筛选、分页、编辑弹窗和权限判断。

这时更合理的结构是：列表和弹窗仍然围绕这个页面内聚，页面专用请求放在页面目录的 `api.ts`，筛选和分页状态放进自定义 hook，数据转换放进 `model.ts`。只有当用户表格组件被订单页、角色页等多个页面稳定使用时，才提升到全局 `components`。

这样做的好处是，修改用户管理页时大部分代码仍在一个页面目录内；真正跨页面复用的部分也有清楚边界。

# Agent 记忆系统：什么时候写回去

## 一句话定义

"写回去"指的是：**一轮对话结束后，从中提取新的信息，写入记忆系统。** 核心原则是写回必须异步执行，不能让用户等待。

## 它解决什么问题

对话中用户会不断透露新信息（个人事实、事件、偏好变化、情绪波动）。如果不及时写入记忆系统，下次对话时 Agent 就无法"记住"这些内容。写入是记忆系统闭环的最后一环。

## 出现前提

- 已完成记忆的存储设计（[[Agent记忆系统-存什么]]）
- 已完成记忆的检索和注入（[[Agent记忆系统-怎么找]]、[[Agent记忆系统-什么时候放进去]]）
- 本轮对话已完成，LLM 已生成回复

## 核心机制

### 最重要的一条：写回必须是异步的

```
用户发消息 → 读取记忆 → 组装 Prompt → LLM 生成 → 返回给用户
                                                      │
                                          ┌───────────┘
                                          ▼
                                   异步写回记忆（用户已经在看回复了）
```

为什么必须异步？因为记忆写入包含三个耗时操作：

1. **信息提取**——用 LLM 从对话中提炼出值得存的信息（额外一次模型调用）
2. **多存储写入**——同时写入向量库、结构化数据库、KV（双写）
3. **画像更新**——合并新旧信息，更新用户画像

如果同步执行，用户要额外等 1-3 秒。"写回去"只影响未来回复，不影响本次回复，所以**绝不能让用户为写入等待**。

在 Cloudflare Workers 环境中的实现：

```javascript
// 返回响应给用户
return new Response(stream);

// 异步写入记忆，不阻塞响应
ctx.waitUntil(
  extractAndWriteMemory(conversation, userId)
);
```

### 写回去的完整流程

#### 第一步：判断"有没有值得存的信息"

不是每轮对话都需要写回。

```
用户："我今天去了一家超好吃的日料店"     → 有新信息，需要提取
用户："嗯嗯"                            → 无新信息，跳过
用户："你真可爱"                        → 偏好信息，可能需要更新亲密度
用户："我今天心情不太好，被老板骂了"     → 事件+情绪，需要提取
```

判断方式：通常用一个轻量级的 LLM 调用（或规则判断），决定是否进入提取流程。

#### 第二步：信息提取

从对话中提炼出值得存的信息，分类输出：

```
本轮对话：
  用户："我最近在学 Rust，感觉生命周期那块特别难"
  Agent："生命周期确实是 Rust 的难点，你现在学到哪部分了？"

提取结果：
{
  "facts": [
    { "key": "learning", "value": "正在学 Rust", "confidence": 0.95 }
  ],
  "events": [
    { "content": "开始学 Rust，觉得生命周期很难", "importance": 0.6 }
  ],
  "emotion": {
    "state": "轻微挫败感",
    "trigger": "学习困难"
  },
  "preferences": [],
  "topic": "Rust 学习",
  "need_write": true
}
```

#### 第三步：冲突检测与合并

写入之前，需要和已有的记忆做冲突检测：

```
已有记忆：{ "key": "city", "value": "杭州" }
新信息：  { "key": "city", "value": "上海" }

→ 不是新增，而是更新（用户搬家了）
```

常见的冲突处理策略：

| 情况               | 处理方式                                   |
| ------------------ | ------------------------------------------ |
| 新事实与旧事实矛盾 | 用新值覆盖旧值（用户信息变了）             |
| 新事实与旧事实互补 | 合并（"杭州" + "前端开发" → 不冲突，都保留） |
| 同一事件的重复提及 | 更新重要性分数，不重复存储                 |
| 旧信息已过时       | 标记过期，从活跃记忆中移除                 |

#### 第四步：双写执行

确定了要写什么之后，同时写入不同的存储介质：

```
提取结果
    │
    ├──→ 向量数据库（事件类：自然语言片段，支持语义检索）
    ├──→ 结构化数据库（事实类：键值对，支持精确查询）
    ├──→ KV 存储（情绪快照、话题上下文，低延迟读取）
    └──→ 用户画像（合并更新，下次对话自动加载）
```

双写的核心挑战是**一致性**——如果向量库写成功了但结构化数据库写失败了，就会出现"语义检索能找到但精确查询找不到"的不一致状态。

解决方案：

```
1. 写入时用事务或批量操作（如 env.DB.batch()）
2. 失败时写入死信队列，定期重试
3. 监控写入成功率，低于阈值告警
```

### 写回去的触发条件

满足任一即触发写入：

```
✓ 对话中出现了新的事实信息（用户透露了个人信息）
✓ 对话中出现了新的事件（有时间和经过）
✓ 用户的情绪发生了变化（从开心变成焦虑）
✓ 话题发生了切换（从旅行聊到工作）
✓ 亲密度发生了变化（正向或负向）

不触发：
✗ 纯寒暄、语气词
✗ 重复已有的信息
✗ 用户只是在确认或纠正 Agent 的回复
```

### 写回去的频率控制

不是每轮对话都做一次完整的提取+写入，那样成本太高：

```
轻量级（每轮）：
  - 情绪快照更新（KV 写入，5-15ms）
  - 话题上下文更新（KV 写入）
  - 亲密度计算

中量级（对话结束时，或每 N 轮）：
  - 事实提取 + 结构化存储写入
  - 事件提取 + 向量库写入
  - 用户画像合并更新

重量级（定期，如每天一次）：
  - 记忆过期清理（删掉过时的、不再相关的记忆）
  - 画像全面校准（检查有没有矛盾的信息）
  - 记忆压缩（把多条碎片化的记忆合并为摘要）
```

### 写回去失败的容错

写入可能失败（数据库超时、网络中断、Worker 执行时间耗尽）：

```
写入失败
    │
    ├── 非关键写入（事件、对话片段）
    │   → 记录日志，不重试。下次对话时重新提取即可
    │
    └── 关键写入（用户画像更新、安全相关信息）
        → 写入死信队列（D1 表），定期重试
        → 监控告警
```

**核心原则：写回失败不应该影响用户体验。** 最坏情况是下次对话时记忆少了一点，而不是当前对话就崩了。

## 收益

- 记忆系统形成闭环：读取 → 使用 → 写入 → 下次读取
- 异步写入不增加用户感知延迟
- 分级频率控制平衡了记忆完整性和系统成本

## 代价

- 信息提取需要额外一次 LLM 调用（成本）
- 双写需要维护多个存储介质的一致性（复杂度）
- 异步写入有极小概率丢失（Worker 执行时间耗尽时）

## 常见误区

| 误区                                             | 正确做法                                           |
| ------------------------------------------------ | -------------------------------------------------- |
| 每轮对话都做完整的提取+写入                       | 分级频率：轻量每轮，中量每 N 轮，重量定期          |
| 同步写入，让用户等写入完成                        | 异步写入，用 waitUntil 不阻塞响应                  |
| 写入失败就报错                                   | 非关键写入静默失败，关键写入走重试队列             |
| 只写入一种存储，忽略双写                          | 不同类型信息写入不同存储，保证检索通道数据完整     |
| 不做冲突检测，新旧信息矛盾时都保留                | 先检测冲突，矛盾时用新值覆盖，互补时合并           |
| 不做定期清理，记忆只增不减                        | 定期过期清理和压缩，避免记忆库膨胀                 |

## 适用场景

- 所有需要将对话中产生的新信息持久化的 Agent 系统
- 异步写入的 Serverless 环境（Cloudflare Workers、Vercel Edge Functions）
- 需要维护记忆一致性的长期对话系统

## 一个真实例子

```
本轮对话：
  用户："我换工作了，现在在字节跳动做后端开发"
  Agent："恭喜你换了新工作！字节的后端技术栈主要用什么呀？"
  用户："Go 为主，还在适应中"
  Agent："Go 上手不难，有问题随时找我聊。"

  ↓ 用户看到回复，对话结束

  ↓ 异步写入开始（ctx.waitUntil）

  ① 判断：有新信息（换工作）→ 触发提取

  ② 提取：
     facts: [
       { key: "job", value: "后端开发", updated: true },      ← 覆盖旧的"前端开发"
       { key: "company", value: "字节跳动", updated: true },
       { key: "tech_stack", value: "Go", new: true }
     ]
     events: [
       { content: "跳槽到字节跳动做后端，用Go，还在适应", importance: 0.8 }
     ]
     emotion: { state: "期待+适应中" }

  ③ 冲突检测：
     job: "前端开发" → "后端开发"（覆盖）
     city: "杭州" → 无变化（保留）

  ④ 双写：
     向量库 ← 事件片段 ✓
     结构化DB ← 事实键值对 ✓
     KV ← 情绪快照 ✓
     画像 ← 合并更新 ✓

  ⑤ 下次对话时：
     Agent 自动知道用户在字节跳动做后端、用 Go、还在适应期
```

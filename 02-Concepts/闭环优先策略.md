# 闭环优先策略

## 一句话定义

闭环优先是AI软件工程落地的核心策略：不逐节点替换人工，而是先建立一个端到端封闭的"AI全流程小闭环"，在闭环内迭代积累偏差案例库和规则，再逐步扩大闭环边界，最终吞并整个软件工程流程。

## 它解决什么问题

逐节点替换（先替换编码、再替换测试、再替换部署）的策略是错的——每个AI节点被人节点切成孤岛，无法形成端到端的AI主导流程。AI在孤立节点中缺乏完整上下文，且没有自动验证回路来纠正自己的错误。闭环优先确保AI在一个完整的、可自验证的子领域内形成自主循环。

## 出现前提

- 已识别出一个"足够小、足够确定"的子领域
- 该子领域有可用的确定性裁判机制
- 团队愿意先聚焦小范围深耕而非急于全面铺开

## 核心机制

**四步路径**：

```
第一步：建立小闭环（挑一个足够小、足够确定的子领域）
第二步：闭环内迭代（积累偏差案例库，沉淀规则）
第三步：扩大闭环边界
第四步：闭环吞并（扩张到整个软件工程流程）
```

**展开顺序**按形式化程度从高到低：

| 节点 | 形式化程度 | 验证可自动化程度 | 当前可建闭环 |
|---|---|---|---|
| 需求分析 | 低（自然语言） | 低 | 困难 |
| 系统设计 | 中 | 中 | 部分可行 |
| 编码+测试 | 高 | 极高 | 最适合先做 |
| 发布运维 | 高 | 高 | 适合做 |

**为什么编码+测试最适合先做**：代码的形式化程度最高——编译通过/失败、测试通过/失败都是客观事实。AI可以在这个闭环内自主完成"生成→验证→修复→再验证"的循环，不需要人介入每一步。

**六阶段演进**（以编码+测试产线为例）：

- **阶段1**：70%需求0人工代码（当前业界主流位置）
- **阶段2**：90%需求0人工代码（最先进团队位置）
- **★ 阶段3（质变点）**：70%需求0人工**接管**——从"概率正确"到"工程可靠"的跨越
- **阶段4**：80-90%需求0人工接管，单元近乎完全自治
- **阶段5**：搭建分工协调总线，连接多个自治单元
- **阶段6**：方法论沉淀为可复制模式，复制到所有研发环节

**阶段2→3是质变**，不靠模型能力升级解决，必须靠：确定性强迫接触（自动验证基础设施）、场景驱动的隐性知识蒸馏、AI主动反思机制。

## 收益

- 避免逐节点替换导致的AI孤岛问题
- 在一个可控范围内快速积累偏差案例和规则
- 方法论可指数加速复制：第一条产线x年 → 第二条x/2年 → 第三条x/4年

## 代价

- 初期看起来"慢"——只做一个小闭环而非全面铺开
- 需要找到合适的"足够小、足够确定"的切入点
- 闭环边界判断需要领域判断力

## 常见误区

- 逐节点替换：编码用AI → 测试用AI → 部署用AI，每个节点之间仍然需要人做"胶水"
- 一开始就试图建全流程闭环——应该从最小的可行闭环开始
- 等模型能力升级来解决阶段2→3的质变——质变不靠模型，靠工程机制

## 适用场景

- 任何想要从Copilot模式转向AI为中心模式的团队
- 编码+测试是最佳切入点
- 发布运维是下一个扩展方向

## 一个真实例子

一个电商团队先选"交易域"建立编码闭环：AI接收"24小时未支付自动取消"的自然语言需求 → AI生成代码 → 编译器+单测验证 → AI修复失败 → 再验证 → 全绿后自动部署灰度。人在这个闭环中只做三件事：定义验收标准、维护验证规则、处理AI反复修不过的偏差case。闭环跑通后，将方法论复制到库存域、商品域、用户域。

# 浏览器事件循环与渲染机制

## 一句话定义

浏览器事件循环与渲染机制，是指浏览器如何在主线程上调度宏任务、微任务、动画回调与渲染流水线，并把 DOM 变化在合适的时机显示到屏幕上的运行方式。

## 它解决什么问题

浏览器需要在单线程 JavaScript、用户输入、页面渲染和异步回调之间安排执行顺序，保证交互可响应、页面能刷新、异步逻辑有确定时机。

## DOM 渲染为什么不会立刻上屏

当 JavaScript 修改 DOM、样式或结构时，浏览器通常不会立刻把变化画到屏幕上，而是等当前 JavaScript 执行结束后，再统一进入布局和绘制。

- `element.style.color = 'red'` 执行后，屏幕不会马上变红
- 连续修改多个 DOM，浏览器通常会合并成一次渲染
- 修改 DOM 后立刻读取布局信息，可能触发强制同步布局，带来性能开销

## 浏览器一帧里的执行顺序

浏览器以约 60fps 刷新屏幕，每帧约 16.6ms。一次可能发生渲染的典型顺序如下：

1. **Input events**：处理 click、scroll、touchmove、keydown 等输入
2. **JS execution**：执行当前宏任务，以及宏任务结束后需要清空的全部微任务
3. **requestAnimationFrame**：在渲染前执行动画回调
4. **Style**：重新计算受影响元素的样式
5. **Layout**：计算元素几何信息
6. **Paint**：生成绘制指令
7. **Composite**：按图层合成后交给 GPU 上屏

### 各步骤的线程归属

| 步骤 | 线程 |
|---|---|
| Input → Layout | 主线程 |
| Paint（记录指令） | 主线程 |
| Raster（画像素） | 光栅化线程池 |
| Composite | 合成线程 |

`transform` 和 `opacity` 动画常被认为性能更好，原因在于它们更容易只经过合成阶段，不需要重新参与样式计算、布局和重绘。

### 时间是怎么消耗的

浏览器不会给每个阶段分配固定时间，而是顺序执行整帧工作。哪一段耗时长，哪一段就会挤占这一帧的预算。

- 渲染阶段常见耗时大约 2-4ms，剩余时间才留给 JavaScript
- JavaScript 长任务如果跑到 50ms，就会直接错过多帧
- 一帧内还有余量时，`requestIdleCallback` 才可能得到空闲窗口

### 不是每帧都会渲染

只有页面变脏时，浏览器才需要进入渲染流水线，比如：

- DOM 改动
- CSS 改动
- 滚动
- 动画持续进行

页面没有可见变化时，浏览器可能直接跳过整段渲染流程。

## 宏任务、微任务与渲染的顺序

核心顺序是：

**一个宏任务 → 清空所有微任务 → 视情况渲染 → 下一个宏任务**

### 具体规则

1. 每次事件循环先取出一个宏任务执行
2. 宏任务结束后，立即清空微任务队列
3. 清空时，执行过程中新增的微任务也要继续执行，直到队列为空
4. 微任务清空后，浏览器才有机会进入渲染阶段
5. 渲染结束后，再取下一个宏任务

```js
setTimeout(() => console.log('timeout'), 0);
Promise.resolve().then(() => console.log('promise'));
console.log('sync');
// 输出：sync → promise → timeout
```

### 为什么每个宏任务结束后都要清空微任务

微任务的作用，是把一段逻辑放到当前执行上下文结束之后、控制权真正交出去之前执行。这样 Promise 链和其他依赖当前状态一致性的逻辑，能尽快连续完成。

如果把 `.then()` 当作宏任务处理，中间可能插入别的宏任务，导致状态变化和额外延迟。

### 常见微任务来源

- `Promise.then() / .catch() / .finally()`
- `await` 之后的代码
- `queueMicrotask(fn)`
- `MutationObserver`
- `process.nextTick()`（Node.js 专属）

### 微任务也会卡住页面

```js
function loop() {
  queueMicrotask(loop);
}
queueMicrotask(loop);
```

只要微任务队列始终清不空，渲染和后续宏任务都会被堵住。

## requestAnimationFrame 为什么更适合动画

`requestAnimationFrame` 的回调发生在渲染前，时机与屏幕刷新同步，所以 DOM 修改更容易在当前这一帧生效。

`setTimeout(fn, 16)` 只是表示至少 16ms 以后进入宏任务队列，真正执行时间还会受到这些因素影响：

- 当前调用栈是否结束
- 前面是否排着其他宏任务
- 最小延迟钳位
- 后台标签页节流

```js
function animate() {
  ball.style.top = y + 'px';
  y += 2;
  requestAnimationFrame(animate);
}
```

如果定时器回调到来时，当前帧刚完成渲染，这次 DOM 修改就只能等到下一帧才会显示。

### rAF 的时机

```text
取一个宏任务 → 清空微任务 → [如果该渲染了] → rAF → Style → Layout → Paint → Composite → 下一个宏任务
```

浏览器不会在每次微任务清空后都执行渲染，只有到达合适的渲染时机并且页面存在可见更新时，才会进入这一段流程。

## Layout Thrashing（布局抖动）

布局抖动指的是在同步代码里反复交替读写布局相关属性，导致浏览器被迫多次执行强制同步布局。

```js
for (let i = 0; i < 100; i++) {
  el[i].style.width = box.offsetWidth + 'px';
}
```

更合适的方式是先批量读取，再批量写入：

```js
const width = box.offsetWidth;
for (let i = 0; i < 100; i++) {
  el[i].style.width = width + 'px';
}
```

### 触发强制布局的典型条件

1. 之前的写操作已经把布局标记为脏
2. 紧接着读取布局相关属性

常见读取项包括：

- `offsetWidth / offsetHeight`
- `clientWidth / clientHeight`
- `scrollTop / scrollHeight`
- `getBoundingClientRect()`
- `getComputedStyle()`

## 长任务超过 16ms 时怎么处理

### 1. 把工作移出主线程

- **Web Worker**：把计算密集型任务放到 Worker
- **OffscreenCanvas**：把 Canvas 渲染控制权转给 Worker
- **WebAssembly**：适合高性能计算逻辑

### 2. 把长任务切成多段

```js
function processChunk(startIndex) {
  const startTime = performance.now();
  let i = startIndex;

  while (i < data.length) {
    processItem(data[i]);
    i++;
    if (performance.now() - startTime > 4) break;
  }

  if (i < data.length) {
    setTimeout(() => processChunk(i), 0);
  }
}
```

常见调度手段：

| 方式 | 特点 | 适合 |
|---|---|---|
| `MessageChannel` | 延迟低 | 尽快恢复的分片 |
| `setTimeout(0)` | 兼容性好 | 通用兜底 |
| `requestAnimationFrame` | 与下一帧对齐 | 和渲染相关的任务 |
| `scheduler.yield()` | 保留调度优先级 | 新浏览器环境 |
| `requestIdleCallback` | 时机不确定 | 非紧急任务 |

### 3. 减少总工作量

| 手段 | 场景 |
|---|---|
| 虚拟列表 | 长列表、表格 |
| 防抖 / 节流 | 搜索联想、滚动回调 |
| 缓存 / Memo | 重复计算 |
| 批量 DOM 操作 | 动态列表渲染 |
| CSS 动画替代 JS 动画 | 位移、缩放、透明度 |

## will-change 的作用

`will-change` 用来提前告诉浏览器，某个元素即将发生某类变化。浏览器可能因此提前准备独立图层，减少动画期间的重绘成本。

```css
.animated-box {
  will-change: transform;
}
```

常见用法是只在变化前短暂开启，用完及时移除，避免长期占用额外显存。

## 事件循环不只属于 JavaScript

事件循环是一种通用并发模型，很多运行时和框架都在使用：

- **Python asyncio**
- **Rust Tokio**
- **Go runtime / netpoller**
- **Java Netty**
- **macOS / iOS RunLoop**
- **Android Looper**
- **Flutter Dart Event Loop**
- **Nginx / Redis**

它们的实现细节不同，但都在解决同一类问题：如何安排任务、事件和回调的执行顺序。

## 常见误区

- **“JS 改了 DOM 就会立刻上屏”**：页面上屏通常要等当前执行结束和下一次渲染时机
- **“setTimeout(16) 就等于 60fps 动画”**：定时器和屏幕刷新不同步
- **“微任务只是更快一点的异步”**：它的关键价值在于时机，发生在当前宏任务之后、渲染之前
- **“不卡顿就说明主线程空闲”**：合成线程可以让部分动画继续流畅，但主线程仍可能已经很忙

# 影子发布与灰度验证

## 一句话定义

一套用于安全替换 AI 管线中可插拔节点的三阶段验证方法论：影子模式验证「不同」，A/B 分桶验证「更好」，渐进发布验证「可稳定上线」。

## 它解决什么问题

AI 管线中的节点（记忆检索、情绪分类、Prompt 模板等）在工程上是可替换的，但「能换」不等于「上线安全」。核心难点：

- 用户输入空间太大，单元测试测不完
- LLM 输出非确定，不能简单 `assertEqual`
- 质量变化是渐进的，不会立刻报错

所以节点替换必须有验证路径，而不是直接全量切换。

## 出现前提

- 系统具备可观测性（Tracing、指标体系）
- 节点已经被设计为可插拔（v1/v2 并存）
- 有稳定的用户标识可用于分桶（userId / sessionId）

## 核心机制

### 阶段一：影子模式（Shadow Mode）

新旧节点同时执行，但只有旧节点参与正式回复；新节点在后台「陪跑」，只用于对比和记录。

关键纪律：
- 用 `.then()` 而不是 `await`，不阻塞主流程
- 必须有 `.catch()`，影子错误绝不能抛到主路径
- 不修改状态、不写正式业务数据

适合回答：新旧版本结果差异大不大、召回条数/平均得分是否提升。
**不适合**回答：用户最终体验是否变好——因为新版本根本没被用于生成回复。

### 阶段二：A/B 分桶对比

把用户分成对照组（旧版）和实验组（新版），对比关键指标。

第一原则：**先稳定，再随机**。

- 用确定性哈希（`hash(userId + experiment) % 100`）而不是 `Math.random()`
- 同一用户在整个实验期必须稳定落在同一组
- 哈希输入必须拼上 `experiment` 名称，让不同实验彼此解耦
- 分组信息必须写入 Trace 的 `input` 字段，否则后期无法归因

关键经验：**不要只看直接指标**。像「单会话消息数」「次日留存」「中途退出率」这种间接指标，往往比技术指标更接近真实体验。

### 阶段三：渐进式发布（Progressive Rollout）

A/B 验证通过后，按 10% → 30% → 50% → 100% 分阶段放量。

本质不是「慢慢上线」，而是**验证实验结论能否从小样本扩展到大样本**。小流量下正常的问题（下游排队、P99 恶化、缓存穿透）只有放量后才暴露。

必须复用同一套确定性哈希，保证：
- **稳定性**：已进入新版的用户不会被切回旧版
- **单调扩张**：扩到 30% 时只是把 0-29 号桶开放，已在 0-9 的用户继续保留

桶位刻度尺类比：0-99 是一条固定刻度，放量就是把 treatment 截断阈值往右推，门越开越大但进门的人不会被赶出来。

### 自动回滚

手动回滚的致命问题是**反应慢**——等发现异常、讨论确认，半小时过去了。正确做法是上线前就定义好自动回滚条件：

- 实验组错误率超过对照组 2 倍 → 回滚
- P95 延迟超过阈值（如 500ms） → 回滚
- 由定时任务（Cloudflare Cron Triggers）每几分钟检查，而不是在每个请求里检查

## 收益 vs 代价

| 收益                         | 代价                      |
| -------------------------- | ----------------------- |
| 节点升级风险可控，出问题时影响范围被限制在一小部分用户 | 需要双写、双执行，短期基础设施成本翻倍     |
| 有数据化证据支撑「新版更好」的判断，不靠直觉      | 需要完善的 Tracing 和指标体系作为前提 |
| 回滚自动化，避免「发现异常时已经太晚」        | 实验设计、阈值定义需要额外的工程投入      |
| 团队敢于频繁改节点，迭代速度反而更快         | 同时跑多个实验时，归因分析复杂度提高      |

## 常见误区

- **「渐进发布是 A/B 分桶的替代品」**：两者回答的是不同问题。A/B 验证「是否更好」，渐进发布验证「能否规模化」。必须是先 A/B 再渐进。
- **「每次放量重新随机分桶」**：会造成样本重洗，观测到的指标变化混杂了「流量增加」和「样本重洗」两种影响，数据污染严重。必须用同一套哈希做单调扩张。
- **「只对 userId 做哈希」**：不同实验会高度耦合，同一用户在所有实验里永远落在同一边。必须把 `experiment` 拼进哈希输入。
- **「只看全局平均指标」**：平均值掩盖局部问题。必须看分层指标（新老用户、高低活跃、地区、设备类型）和存量 vs 增量。
- **「回滚交给人临场判断」**：半小时的讨论窗口往往就是事故扩大的窗口。必须上线前就定义好自动回滚条件。
- **「影子模式能验证体验是否变好」**：不能。影子结果没进入正式回复，只能回答「结果是否不同」，回答不了「体验是否更好」。

## 一个真实例子

记忆检索节点从 v1 升级到 v2：

1. **影子模式**：用户请求时，v1 正常返回结果；v2 在后台用 `.then()` 跑，记录两个版本的召回重合度（overlap）。
2. **A/B 分桶**：`hash("user_123:memory_retrieval_v2") % 100 < 10` 的用户进入实验组。观测发现 treatment 组记忆命中率 75% vs control 68%、单会话消息数 52 vs 45（间接指标更有力）——但平均检索耗时从 95ms 升到 120ms。
3. **渐进发布**：从 10% → 30% → 50% → 100%，每档观察错误率、P95 延迟、降级率。扩到 50% 时发现增量用户出现 P99 尖刺，说明问题和规模相关，暂停放量排查。
4. **自动回滚**：Cron Trigger 每 5 分钟检查一次，v2 错误率超过 v1 的 2 倍或 P95 超过 500ms 时自动切回 v1。

## 适用场景

- AI 管线中的节点升级（检索、分类、生成、模板）
- 效果变化是渐进的、无法用单元测试断言的改动
- 有足够真实流量支撑统计显著性的在线系统
- 需要跨多轮对话/长周期观察的系统（AI 伴侣、推荐系统）

## 不适用场景

- 单元测试就能验证的确定性改动（bug 修复、类型调整）
- 用户量极小、样本量不足的早期项目——A/B 结论不显著
- 没有可观测性基础设施的项目——无从对比
- 一次性迁移、不可逆的架构调整（如数据库 schema 变更）

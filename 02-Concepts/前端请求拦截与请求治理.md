# 前端请求拦截与请求治理

## 一句话定义

前端请求拦截与请求治理，是指在真实网络请求发出前后，用统一入口处理缓存、重复请求取消、失败重试和触发频率控制，避免每个页面各自处理同一类请求问题。

## 它解决什么问题

页面直接调用接口时，常见问题不是“能不能发请求”，而是请求行为是否可控：

- 搜索输入、筛选条件变化时，同一个接口被短时间重复触发
- 用户快速切换页面或条件，旧请求比新请求更晚返回，导致页面显示过期数据
- 同一参数的请求反复命中网络，浪费带宽和等待时间
- 临时网络失败后，业务代码需要到处手写重试逻辑
- 组件卸载或请求结束后，取消控制器、缓存和 pending 状态没有被清理

这类问题如果分散在组件里处理，页面代码会越来越重，也很难保证不同页面行为一致。

## 出现前提

- 前端项目已经有多个页面或组件共享同一套请求能力
- 接口调用不再只是简单 `get` / `post`，还需要处理缓存、取消、重试等横切逻辑
- 用户交互会高频触发请求，例如搜索、筛选、分页、表单校验
- 项目运行环境可能包含浏览器、SSR、测试环境，需要避免直接假设 `window` 一定存在

## 核心机制

### 1. 用统一入口包住底层请求库

常见结构是两层：

| 层级 | 职责 |
|---|---|
| 外层 `request()` | 合并默认配置，处理防抖，决定是否真正创建请求 |
| 内层请求实例 / 拦截器 | 处理缓存命中、重复请求取消、响应写缓存、失败重试 |

防抖适合放在请求创建之前，因为它的目标是“这次请求要不要创建”。缓存、取消和重试适合放在请求实例或拦截器中，因为它们依赖已经确定的请求 key、配置和响应结果。

### 2. 用 request key 识别同一类请求

请求治理通常需要先把 URL、method、params、data 等信息组合成稳定 key。这个 key 决定：

- 缓存能不能命中
- 新请求是否要取消旧请求
- pending 状态应该写入和清理哪一条记录
- 防抖计时器是否属于同一类请求

key 的设计要稳定且足够区分业务语义。过粗会误伤不同请求，过细会导致缓存和取消失效。

### 3. pendingMap 管理正在进行中的请求

`pendingMap` 的核心作用，是记录“哪些请求还没结束，以及怎么取消它们”。常见记录内容包括取消函数和递增的 requestId。

requestId 的价值在于区分新旧请求：当新请求进来时，只取消更旧的同 key 请求；当响应返回时，也只清理属于自己的 pending 记录，避免把更新的请求状态误删。

### 4. 缓存命中也要清理 pending 状态

缓存命中后通常会直接返回结果，不再进入后续网络请求流程。如果这时不清理 pending 状态，旧的取消控制器和请求记录可能继续留在内存中，后续请求也可能误以为仍有同类请求正在进行。

判断规则是：缓存命中意味着这次不再需要网络请求，也就不应该继续保留 pending 语义。

### 5. 浏览器环境判断要用 `typeof window`

在 SSR、Node.js 测试、CLI 脚本或 Worker 环境中，`window` 可能根本不存在。此时直接写 `if (!window)` 会因为变量未声明而报错。

更安全的写法是：

```ts
if (typeof window !== 'undefined') {
  // browser only
}
```

只要变量可能不存在，就应该用 `typeof` 做存在性判断。

### 6. 防抖和节流解决的是不同问题

| 机制 | 行为 | 适合场景 |
|---|---|---|
| 防抖 debounce | 连续触发时，只执行最后一次 | 搜索输入、表单校验 |
| 节流 throttle | 固定时间窗口内最多执行一次 | 滚动监听、窗口 resize、拖拽 |

请求搜索更常用防抖，因为用户真正关心的是停下输入后的最终关键词。滚动和拖拽更常用节流，因为它们需要持续但受控地更新。

## 收益

- 页面组件不需要重复处理取消、缓存、重试和防抖
- 高频交互下能减少无效请求
- 旧请求被取消后，不容易覆盖新请求结果
- 临时失败可以通过统一重试策略恢复
- 请求行为集中配置，后续修改影响面更可控

## 代价

- 请求模块会变复杂，需要维护缓存、pending、计时器和重试状态
- key 生成规则一旦设计不好，会出现误取消、误缓存或缓存失效
- 自动重试可能放大服务端压力，尤其是接口本身已经异常时
- 防抖会增加响应延迟，不适合所有请求
- 统一封装过厚时，具体业务很难判断请求到底经历了哪些处理

## 常见误区

- **“拦截器可以处理所有请求问题”**：防抖应该发生在请求创建之前，拦截器更适合处理已经创建的请求。
- **“缓存命中就不用管 pendingMap”**：缓存命中跳过了后续流程，更需要主动清理旧 pending 状态。
- **“取消请求只要按 URL 判断就够了”**：method、params、data 往往也会影响请求语义。
- **“失败重试越多越稳定”**：重试只能处理短暂失败，不能掩盖接口持续异常。
- **“防抖和节流可以混用”**：二者控制策略不同，选错会直接影响交互体验。

## 适用场景

- 搜索框、筛选器、分页器会频繁触发同一接口
- 页面切换或条件切换时，需要避免旧请求覆盖新结果
- 多个页面共享同一套接口调用规范
- 某些查询接口适合短时间缓存
- 接口存在偶发失败，需要有限次数自动重试

## 不适用场景

- 写操作、支付、下单等不能被缓存或自动重复执行的请求
- 强实时数据，不允许返回缓存结果
- 请求频率本来很低，封装带来的复杂度大于收益
- 服务端已经提供了更清晰的幂等、缓存和限流能力，前端只需要轻量适配

## 一个真实例子

一个商品搜索页在用户输入关键词时请求列表。用户连续输入 `react`，如果每次输入都立即请求，会产生多次无效请求，并且旧请求可能在新请求后返回。

更稳妥的做法是：外层 `request()` 对同一搜索 key 做防抖，只在输入停止后创建最后一次请求；请求进入拦截器后，用 key 检查是否有旧请求还在进行，如果有就取消旧请求；响应成功后写入短时间缓存，用户来回切换筛选条件时可以直接复用结果。

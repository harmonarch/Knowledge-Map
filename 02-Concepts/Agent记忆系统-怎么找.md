# Agent 记忆系统：怎么找

## 一句话定义

存储解决了"信息在哪儿"的问题，但信息存了找不到等于没存。"怎么找"解决的是：**用户说了一句话，怎么从海量记忆中把最相关的几条捞出来。**

## 它解决什么问题

传统搜索是用户主动搜，知道自己要找什么。但 Agent 的记忆检索不一样——用户说了一句话，Agent 要**自己判断**需要哪些记忆，用户不会明确告诉你"去帮我查一下我之前说过什么"。

所以"怎么找"分两步：

```
第一步：查询理解 —— 用户说了什么？需要什么类型的记忆？
第二步：检索执行 —— 去哪里找？怎么找？
```

## 核心机制

### 第一步：查询理解

用户说了一句话，先判断这句话需不需要去翻记忆。

```
用户："今天天气怎么样"         → 不需要记忆，直接调用天气工具
用户："我之前跟你聊过什么来着"  → 需要翻记忆，语义检索
用户："我的猫叫什么来着"       → 需要翻记忆，精确查询
用户："我们上次聊到哪了"       → 需要翻记忆，时序检索
用户："帮我推荐个餐厅"         → 可能需要偏好记忆
```

查询理解的输出是一个**检索计划**：

```json
{
  "need_memory": true,
  "query_type": ["semantic", "structured"],
  "query_text": "猫的名字",
  "time_range": null,
  "keywords": ["猫", "名字"]
}
```

这一步很关键，但也很脆弱——如果查询理解判断错了，后面的检索质量就会很差。所以需要**降级策略**：理解失败时退回全通道检索。

### 第二步：四种检索通道

#### 通道 1：语义检索

**原理**：把用户的查询和存储的记忆都转成向量（Embedding），计算余弦相似度，取最接近的几条。

```
用户说："之前聊过旅行的那些事"

    ↓ Embedding

查询向量: [0.12, -0.34, 0.56, ...]

    ↓ 与向量库中所有记忆计算相似度

记忆A: "周末去西湖骑行了，特别开心"      相似度 0.87  ✓
记忆B: "打算下个月去成都旅游"             相似度 0.91  ✓
记忆C: "我养了一只橘猫叫小橘"            相似度 0.23  ✗
记忆D: "上周加班到凌晨三点"              相似度 0.18  ✗
```

- **擅长**：模糊的、概念性的、换个说法也能找到（用户说"出去玩"，能找到"去西湖骑行"）
- **不擅长**：精确匹配（用户问"我的猫叫什么"，可能返回一堆关于猫的聊天记录，但精确名字反而不在第一条）
- **技术栈**：Embedding 模型 + 向量数据库（如 CloudFlare Vectorize）

#### 通道 2：精确查询

**原理**：直接对结构化数据库做字段匹配，不走语义。

```
用户问："我的猫叫什么来着"

    ↓ 查询理解判断为精确事实查询

SELECT value FROM user_facts WHERE key = 'pet'

    ↓ 结果

"橘猫，名字叫小橘，三岁"
```

- **擅长**：明确的事实性查询（"我生日几号"、"我在哪个城市"、"我对什么过敏"）
- **不擅长**：模糊查询（用户说"我之前好像说过一个什么来着"——没有明确关键词，精确查询无能为力）
- **技术栈**：KV 存储 / 关系型数据库（如 CloudFlare D1）

#### 通道 3：时序检索

**原理**：按时间戳索引，支持时间维度的查询。

```
用户问："上周我们聊了什么"

    ↓ 时间范围过滤

SELECT * FROM memories
WHERE timestamp BETWEEN '2026-04-07' AND '2026-04-13'
ORDER BY importance DESC
LIMIT 10

    ↓ 结果

4月8日："项目赶工很累"
4月10日："周末去西湖骑行了"
4月12日："开始计划成都旅行"
```

- **擅长**：有时间锚点的回忆（"昨天"、"上周"、"最近一个月"）
- **不擅长**：没有时间线索的查询（用户更多时候直接接着话题聊）
- **技术栈**：带时间索引的数据库

#### 通道 4：用户画像召回

**原理**：不是用户触发的检索，而是**每次对话自动加载**的全局画像。

```
用户画像（每次对话自动注入 Prompt）：
┌─────────────────────────────┐
│ 名字：未知                    │
│ 城市：杭州                    │
│ 职业：前端开发                 │
│ 宠物：橘猫 小橘 三岁           │
│ 偏好：简短回复、喜欢 emoji     │
│ 近期状态：工作压力大           │
│ 亲密度：6.5                   │
└─────────────────────────────┘
```

与其他通道的区别：
- 通道 1-3 是**按需检索**——用户说了什么，去找相关的记忆
- 通道 4 是**主动加载**——不管用户说什么，这些信息每次都在

### 四个通道怎么配合：混合检索

单个通道都有盲区，生产级系统是**多通道并行，结果融合**：

```
用户说："最近有没有什么开心的事"

        ↓ 查询理解

    ┌─────────────── 并行检索 ───────────────┐
    │                                        │
    │  语义检索 → "西湖骑行特别开心" (0.89)   │
    │  时序检索 → 近一周的重要事件             │
    │  画像加载 → 用户近期状态: 工作压力大     │
    │                                        │
    └──────────── ↓ 结果融合 ────────────────┘
                    │
                    ▼
            综合排序后注入 Prompt
```

#### 结果融合（Reranking）

多条检索结果可能重叠、矛盾或无关，需要一个策略来决定最终给 LLM 看哪几条：

| 维度       | 含义                       | 权重示例 |
| ---------- | -------------------------- | -------- |
| 相似度     | 语义匹配程度               | 0.3      |
| 重要性     | 存储时标记的分数           | 0.3      |
| 时效性     | 越近越相关                 | 0.2      |
| 类型匹配   | 查询类型与记忆类型是否匹配 | 0.2      |

融合后，按 Token 预算截断——prompt 里给记忆留了多少 Token，就只取排名最高的几条。

### 检索不到怎么办：降级策略

```
查询理解成功 → 走对应的精确检索通道
     │
查询理解失败 → 退回全通道并行检索（宁可多找，不要漏掉）
     │
全通道都没找到 → 不注入记忆，LLM 正常回复
     │
找到了但分数很低 → 注入但标记为 degraded（质量存疑）
```

degraded 状态是检索系统最需要警惕的情况——"流程还能继续，但已进入次优路径"。检索质量下降了，没有报错，用户不会察觉，但体验会慢慢变差。

## 收益

- 不同类型的查询走不同的最优通道，检索精准度远超单一方式
- 多通道并行覆盖各自的盲区
- 结果融合确保注入 Prompt 的记忆既相关又精炼

## 代价

- 查询理解本身是一次 LLM 调用，增加延迟和成本
- 多通道并行检索增加系统复杂度
- Reranking 策略需要调参，没有万能公式
- 降级场景的监控和排查需要完善的 Tracing

## 常见误区

| 误区                           | 正确做法                                       |
| ------------------------------ | ---------------------------------------------- |
| 所有查询都走语义检索           | 精确事实走结构化查询，更快更准                  |
| 查询理解失败就放弃检索         | 降级到全通道检索，宁可多找不要漏掉              |
| 检索结果直接全部塞入 Prompt    | 按 Token 预算截断，只留最相关的几条             |
| 认为检索到就够了，不关注分数   | 低分结果标记 degraded，监控检索质量             |
| 用户画像也走检索通道           | 画像是每次自动加载的，不走检索                  |

## 适用场景

- 所有需要从记忆库中检索信息的 Agent 系统
- 任何对话中需要"回忆"之前聊过什么的场景
- 用户画像驱动的个性化回复

## 一个真实例子

```
用户："上次我说想去的那个地方是哪来着？"

  ① 查询理解：需要记忆，类型为语义+时序
     query_type: ["semantic", "temporal"]
     time_range: "recent"

  ② 并行检索：
     语义检索 → "打算下个月去成都旅游" (0.88)
     时序检索 → 近两周重要事件列表

  ③ 结果融合：成都旅行排名最高

  ④ 注入 Prompt：
     "用户之前提到想去成都旅游"

  ⑤ LLM 回复：
     "你上次说想去成都，打算什么时候出发呀？"
```

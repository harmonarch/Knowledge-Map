# Monorepo

## 一句话定义

将多个相关项目放在同一个代码仓库中管理的工程策略。

## 出现前提

当多个子系统之间存在大量共享代码（类型定义、Schema、工具函数）且需要频繁同步变更时，多仓库的版本协调成本过高，需要一种能实现原子提交和直接代码引用的仓库组织方式。

## 核心机制

- **Workspace 协议**：通过包管理器（yarn/pnpm）的 workspaces 功能，让仓库内的多个子项目互相引用，无需发布到 npm
- **任务编排**：通过工具（Turborepo/Nx）管理多项目的构建顺序、并行执行和缓存
- **原子提交**：一次 commit 可以同时修改多个子项目，保证版本一致性

## 收益 vs 代价

| 收益                        | 代价                    |
| ------------------------- | --------------------- |
| 类型/Schema 直接 import，零延迟同步 | 仓库体积较大                |
| API 变更一个 PR 搞定，不存在版本时间差   | 权限控制较粗，需要 CODEOWNERS  |
| 一份工具链配置，全局统一              | CI/CD 需要按变更范围触发，配置稍复杂 |
| 本地开发一条命令启动所有服务            | 新人上手需要理解整体结构          |

## 常见误区

- **"Monorepo = 巨石应用"**：Monorepo 是仓库组织方式，不是架构模式。子项目仍然独立构建、独立部署
- **"小项目不需要 Monorepo"**：关键不是项目大小，而是子系统之间是否有强类型依赖和频繁的同步变更需求
- **"Monorepo 会让仓库变得很慢"**：现代工具（Turborepo/Nx）通过增量构建和缓存解决了这个问题，只有 Google/Meta 级别的超大仓库才会遇到真正的性能瓶颈

## 适用场景

- TypeScript 全栈项目，前后端需要端到端类型安全
- 多个子系统共享 Zod Schema、错误码、工具函数
- 小型团队，追求快速迭代和即时反馈
- 子系统部署目标不同但代码高度耦合（如 Hono → Workers, Next.js → Vercel）

## 不适用场景

- 子系统使用完全不同的语言/技术栈
- 大型团队需要严格的仓库权限隔离
- 子系统之间几乎没有共享代码
- 各子系统发布节奏完全独立

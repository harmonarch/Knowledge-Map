# React 并发模式与调度机制

## 一句话定义

React 并发模式与调度机制，是指 React 把一次完整更新拆成可中断的小段，并按优先级安排执行顺序，从而让用户交互优先、长任务分批完成的一套运行方式。

## 它解决什么问题

前端界面更新经常要和输入事件、浏览器渲染、异步回调共享同一条主线程。只要一次渲染任务太长，页面就会卡住，输入反馈也会变慢。React 需要一种机制，既能推进整棵组件树的更新，又能在合适时机让出执行权，让高优先级交互先被处理。

## 出现前提

- 组件树更新可能很大，单次同步跑完会堵住主线程
- 用户输入、浏览器渲染和 React 渲染要竞争同一段执行时间
- 更新之间存在轻重缓急，搜索输入和过渡列表不该同等对待
- React 需要暂停后继续推进未完成的渲染进度，而不是每次都从头递归跑完整棵树

## 核心机制

1. **时间切片**：React 给自己设置较短的执行预算，典型实现里每轮只工作约 5ms，时间到了就先让出控制权。
2. **Fiber 可中断遍历**：组件树不再依赖不可暂停的递归调用栈，而是用 Fiber 链表和 `workInProgress` 指针保存当前进度，下一轮可以从中断点继续。
3. **任务分层调度**：进入调度器队列的任务，不是单个 Fiber 节点，而是“一次完整的根更新”；任务内部再逐个处理 Fiber 单元。
4. **优先级驱动执行**：不同更新会被映射成不同优先级，高优先级更新可以打断低优先级工作，低优先级任务晚些继续或重做。
5. **批处理合并更新**：多次 `setState` 先通过微任务聚合，再统一推进根节点调度，减少重复进入渲染流程。
6. **宏任务续跑机制**：调度器借助 `MessageChannel` 这类低延迟宏任务，把未完成工作一轮轮接着跑，直到当前队列清空。

## 收益

- 长列表、重计算和复杂组件树更新更不容易把页面直接卡死
- 用户点击、输入这类交互能更早得到反馈
- React 可以在保持单线程模型的前提下实现“分段完成大任务”
- 批处理减少了重复调度和中间态渲染
- 优先级机制给 `startTransition`、`useTransition` 这类 API 提供了语义基础

## 代价

- 渲染流程明显比同步递归复杂，理解门槛更高
- 低优先级工作被打断后，部分已完成结果可能被丢弃并重做
- 调试时看到的阶段性界面，不一定是最终状态
- 更新链路跨越事件循环、调度器和 Fiber 三层结构，定位问题更费劲
- “任务”和“更新单元”是两层概念，容易在阅读源码时混淆

## 常见误区

- **“并发就是多线程并行”**：React 这里主要是在单线程上分时调度，不代表组件树在多个线程同时计算
- **“时间切片就是把 16.6ms 一帧硬切给 React”**：React 只拿其中一小段预算，因为同一帧里还有事件和浏览器渲染要运行
- **“进入队列的是单个 Fiber”**：调度器队列里装的是根任务，Fiber 级遍历发生在任务内部
- **“被中断的低优先级工作肯定能无损续跑”**：遇到高优先级插队时，低优先级阶段结果可能作废并重做
- **“startTransition 会让更新变快”**：它做的是降级优先级，让紧急交互先执行，不是直接提升吞吐

## 一个真实例子

原始材料里有一个搜索场景：用户输入关键字时，输入框内容更新需要立刻反馈，搜索结果列表的重算可以稍后完成。把结果更新包进 `startTransition(() => setResults(search(value)))` 之后，React 会把这部分工作标成低优先级任务。这样输入框更新能先响应，结果列表则在后续调度轮次里逐步完成，必要时还可以被新的高优先级输入再次打断。

这个例子说明，并发模式追求的不是“尽快把所有工作一次做完”，而是“先把当前最影响交互体验的部分做出来”。

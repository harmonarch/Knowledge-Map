# React 并发模式与调度机制

## 一句话定义

React 并发模式与调度机制，是指 React 把一次完整更新拆成可中断的小段，并按优先级安排执行顺序，从而让用户交互优先、长任务分批完成的一套运行方式。

## 它解决什么问题

前端界面更新经常要和输入事件、浏览器渲染、异步回调共享同一条主线程。只要一次渲染任务太长，页面就会卡住，输入反馈也会变慢。React 需要一种机制，既能推进整棵组件树的更新，又能在合适时机让出执行权，让高优先级交互先被处理。

## 出现前提

- 组件树更新可能很大，单次同步跑完会堵住主线程
- 用户输入、浏览器渲染和 React 渲染要竞争同一段执行时间
- 更新之间存在轻重缓急，搜索输入和过渡列表不该同等对待
- React 需要暂停后继续推进未完成的渲染进度，而不是每次都从头递归跑完整棵树

## 核心机制

1. **时间切片**：React 给渲染任务设置较短的执行预算，典型实现里每轮只工作约 5ms，时间到了就先把主线程还给浏览器。
2. **中断与恢复**：`workLoopConcurrent` 每处理完一个 Fiber 都会检查 `shouldYield()`，超时就暂停，下一轮从当前 `workInProgress` 位置继续。
3. **插队与重做**：高优先级更新到来时，当前低优先级渲染可能直接作废，从 `root.current` 重新 fork 一棵新的 `workInProgress` 树开始计算。
4. **两层调度模型**：外层 Scheduler 决定“先做哪个 root 的任务”，内层 Reconciler 决定“这次任务里逐个处理哪些 Fiber 节点”。
5. **Lane 优先级模型**：更新先落到 Lane 位掩码，再映射到 EventPriority 和 SchedulerPriority，用位运算完成合并、筛选和取最高优先级。
6. **双缓冲 Fiber 树**：React 同时维护 `current` 与 `workInProgress` 两棵树，render 阶段只构建新树，commit 阶段再通过指针切换把结果一次性交到屏幕上。
7. **批处理与续跑**：同一轮事件里的多次 `setState` 会先合并，再由 `MessageChannel` 驱动调度器在后续宏任务里持续推进未完成工作。
8. **饥饿防护**：低优先级任务长期得不到执行时，会因过期时间到达而升级，避免一直被更高优先级更新压住。

## 收益

- 大型组件树更新更不容易长期堵住主线程
- 输入、点击这类交互能先拿到反馈
- React 可以把一次根更新拆到多个宏任务里逐步完成
- `startTransition`、`useTransition` 这类 API 有了明确的调度语义
- 批处理和优先级映射减少了重复调度与无意义中间态

## 代价

- 调度、渲染、优先级三层结构叠在一起，源码理解门槛明显提高
- 低优先级 render 可能做到一半被废弃，产生额外重算成本
- 阶段性结果并不稳定，调试时更容易误判“为什么刚算过又重来一遍”
- 任务、Lane、Fiber 单元分别处在不同层级，概念很容易串线
- 并发更新体验更好，但问题定位路径也更长

## 常见误区

- **“并发就是多线程并行”**：React 主要还是单线程分时调度，重点是让出执行权，不是开多个线程同时算组件树。
- **“插队和中断是一回事”**：中断是时间片用完后暂停并续跑，插队是高优先级更新到来后直接废弃当前低优先级 render。
- **“进入 Scheduler 队列的是 Fiber 节点”**：队列里是 root 级 task，Fiber 级遍历发生在 task.callback 内部。
- **“时间切片一定精确等于 5ms”**：`shouldYield()` 只在 Fiber 单元之间检查，所以实际一轮执行时间通常是约 5ms。
- **“低优先级任务可以无限等”**：React 会通过过期时间把拖太久的任务升级，避免彻底饿死。
- **“双缓冲就是每次新建整棵树”**：真正发生的是 alternate 节点复用和 commit 时的指针切换，不是简单删旧建新。

## 一个真实例子

原始材料里给了一个典型场景：用户边输入搜索词，边触发结果列表重算。输入框更新属于高优先级交互，`startTransition` 包裹的结果列表更新属于较低优先级任务。这样 React 会先保证输入反馈，再在后续时间片里推进列表 render；如果用户继续输入，之前尚未完成的低优先级 render 还可能被新的高优先级更新直接插队重做。

这个例子说明，并发调度关注的是交互顺序与主线程占用，而不是一次把所有工作尽快跑完。

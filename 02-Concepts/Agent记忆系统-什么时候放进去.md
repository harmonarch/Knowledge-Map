# Agent 记忆系统：什么时候放进去

## 一句话定义

"放进去"指的是：**检索到的记忆，在什么时机、以什么形式注入到 Prompt 里。** 这不是一个简单的"检索到了就塞进去"，而是一个受 Token 预算约束的**调度决策**。

## 它解决什么问题

Prompt 不是无限长的。模型的上下文窗口有上限（8k~128k），记忆能占的 Token 是有限的、需要和其他内容竞争的。所以需要一个调度机制来决定：什么时候注入、注入什么、注入多少。

## 出现前提

- 已完成记忆的存储（[[Agent记忆系统-存什么]]）
- 已完成记忆的检索（[[Agent记忆系统-怎么找]]）
- 检索结果已经拿到，等待注入 Prompt

## 核心机制

### 硬约束：Token 预算分配

Prompt 的 Token 预算是有限的，需要合理分配：

```
总预算：128,000 tokens
    │
    ├── System Prompt（角色设定、规则、安全指令）     ≈ 2,000 tokens
    ├── 用户画像（自动加载，每次都有）                ≈ 500 tokens
    ├── 记忆注入（检索结果）                        ≈ 1,000-3,000 tokens
    ├── 当前对话历史（最近几轮）                      ≈ 2,000-5,000 tokens
    ├── 工具调用结果（如果有的话）                    ≈ 500-2,000 tokens
    │
    └── 预留给 LLM 生成的空间                      ≈ 剩余全部
```

记忆能占的空间是**动态的**，取决于其他组件占了多少。

### 放进去的时机：Prompt 组装环节

在编排管线中，记忆注入的时机非常明确：

```
安全检查 → 查询理解 → 并行检索/情绪读取 → 【Prompt 组装】 → LLM 生成
                                              ↑
                                         记忆在这里注入
```

Prompt 组装节点负责把所有组件拼成一个完整的 Prompt：

```json
{
  "system": "你是用户的AI伴侣，温柔体贴...",
  "profile": "用户在杭州，养了一只橘猫叫小橘...",
  "memory": "上次聊到成都旅行计划，用户工作压力大...",
  "emotion": "当前情绪：平静，亲密度：6.5",
  "history": "最近3轮对话...",
  "query": "用户当前输入"
}
```

**关键原则：检索必须在 Prompt 组装之前完成。** 如果顺序反了，LLM 拿到的就是一个没有记忆上下文的 Prompt。

### 放进去的优先级

Token 预算不够时，不是所有检索到的记忆都能放进去。需要一个优先级排序：

```
优先级从高到低：

1. 安全相关       → 最高优先，不能省（比如用户有自伤倾向的历史）
2. 当前话题上下文  → 直接影响回复连贯性
3. 用户画像       → 每次都在，但可以压缩
4. 近期情绪状态    → 影响回复语气
5. 语义检索结果    → 按相似度分数取 top-N
6. 历史对话摘要    → 预算有剩余才放
```

预算紧张时从底部开始砍：

```
预算充足时：全部放
预算紧张时：砍掉历史对话摘要
预算很紧时：只保留安全 + 话题 + 画像
预算极紧时：只保留安全 + 画像（最小集）
```

这就是"内存调度"——和操作系统的内存管理一样，物理内存不够时，按优先级决定哪些进程留在内存里，哪些换出去。

### 放进去的形式：不是原文照搬

记忆不是直接把存储的原文粘贴进 Prompt，而是要根据当前上下文做格式化：

#### 事实类记忆 → 压缩为标签

```
存储原文："我养了一只橘猫，叫小橘，三岁了"
注入形式："用户养了一只三岁的橘猫叫小橘"  （一句话概括）
```

#### 事件类记忆 → 保留关键要素

```
存储原文："今天项目终于上线了，熬了两周终于搞完"
注入形式："近期事件：项目上线（两周赶工，4月13日）"  （保留事件+情绪+时间）
```

#### 情绪记忆 → 转化为行为指引

```
存储原文：{ "emotion": "焦虑", "intensity": 0.7, "trigger": "工作压力" }
注入形式："用户近期情绪偏焦虑（工作压力），回复时注意语气柔和，避免追问敏感话题"
```

#### 偏好记忆 → 转化为生成约束

```
存储原文：{ "reply_style": "简短", "emoji_preference": true }
注入形式："回复要求：简短，可以使用emoji"
```

**核心思路：存储是为了完整记录，注入是为了高效利用。** 给 LLM 看的应该是精炼过的、可直接指导生成的信息，而不是原始数据。

### 两种注入模式

不是所有记忆都是"用户问到了才去翻"的，有两种模式：

| 模式         | 触发方式           | 内容                       | 占预算策略             |
| ------------ | ------------------ | -------------------------- | ---------------------- |
| **主动加载** | 每次对话自动注入   | 用户画像、情绪状态、偏好   | 固定占用，预留 Token   |
| **按需检索** | 查询理解触发       | 相关事件、历史对话片段     | 动态占用，按优先级截断 |

主动加载的内容不管用户说什么都在，所以必须精炼，控制在 500 Token 以内。按需检索的内容根据查询动态注入，有多少空间就放多少。

## 收益

- 在有限的 Token 预算内最大化记忆的信息密度
- 优先级机制确保最重要的信息不会因为预算紧张而被丢弃
- 格式化注入让 LLM 能直接利用记忆，而不是面对原始数据

## 代价

- Prompt 组装节点增加了系统复杂度
- 优先级排序和格式化逻辑需要维护和调优
- Token 预算紧张时不得不丢弃部分记忆，可能导致回复质量下降

## 常见误区

| 误区                                       | 正确做法                                             |
| ------------------------------------------ | ---------------------------------------------------- |
| 检索到的记忆直接全部塞进 Prompt             | 按 Token 预算截断，只留最相关的几条                  |
| 把记忆放在 System Prompt 最前面             | 放在靠近用户输入的位置，LLM 对后面的内容注意力更强   |
| 每次都注入全部记忆，不管用户问了什么        | 根据查询理解的结果，只注入相关的记忆                 |
| 原文照搬存储内容                            | 格式化为精炼的、可直接指导生成的信息                 |
| 觉得 Token 预算足够，不控制记忆注入量      | 记忆越多 LLM 注意力越分散，质量反而下降              |

## 适用场景

- 所有需要将检索到的记忆注入 Prompt 的 Agent 系统
- Token 预算有限、需要精细调度的对话场景
- 多类型记忆混合注入的复杂 Agent

## 一个真实例子

```
用户："最近有什么开心的事吗？"

  ① 之前步骤：查询理解 → 并行检索 → 拿到3条记忆

  ② Prompt 组装（Token 预算紧张，只能放2条）：

     按优先级排序：
     ✗ "用户偏好简短回复"（优先级5，被砍）
     ✓ "4月10日去西湖骑行，特别开心"（优先级2，话题相关）
     ✓ "情绪：近期偏焦虑，工作压力"（优先级4，影响语气）

  ③ 格式化后注入 Prompt：
     "近期事件：用户周末去西湖骑行很开心。
      情绪状态：近期偏焦虑（工作压力），语气需温和。
      回复要求：简短。"

  ④ LLM 回复：
     "想起你上周末去西湖骑行玩得很开心呀～这周末要不要再出去放松一下？🚲"
```

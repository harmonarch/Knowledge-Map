# Agent 记忆系统：存什么

## 一句话定义

不是所有对话内容都值得存。"存什么"本质上是一个**信息提炼**过程——从对话中提取出"以后会用到的信息"，扔掉"用不到的噪音"。

## 它解决什么问题

LLM 本身是无状态的，每次调用都是一张白纸。但 Agent 需要跨会话的连续性——记住用户是谁、聊过什么、喜欢什么。

直接把所有对话历史塞给 LLM 行不通：
- **Token 有上限**（8k~128k），长期聊天远超窗口
- **Token 越多成本越高、延迟越大**
- **模型不区分"重要"和"闲聊"**，会随机记住或忘记

所以需要从对话中**提炼**出值得持久化的信息。

## 核心机制

Agent 需要存储 6 类信息：

### 1. 事实类信息

用户在对话中透露的关于自己的客观事实。

```
用户：我养了一只橘猫，叫小橘，三岁了
用户：我在杭州工作，做前端开发
用户：我对花生过敏
```

- **为什么重要**：用户画像的基石。"记得"用户的猫叫小橘，比"记得"用户说了句"嗯嗯"有价值得多
- **存储方式**：结构化键值对，存入数据库（KV / 关系型），支持精确查询

```json
{ "key": "pet", "value": "橘猫，名字叫小橘，三岁" }
{ "key": "city", "value": "杭州" }
{ "key": "job", "value": "前端开发" }
{ "key": "allergy", "value": "花生" }
```

- **检索场景**：用户问"我养了什么猫来着？"——精确查询直接命中

### 2. 事件类信息

用户经历的具体事件，有时间、有经过、有情绪。

```
用户：今天项目终于上线了，熬了两周终于搞完
用户：周末和朋友去西湖骑行了，特别开心
用户：昨天和领导吵了一架，心情很差
```

- **为什么重要**：建立情感连接的关键。记住"上周项目上线很累"，下次问"项目怎么样了"就有上下文
- **存储方式**：自然语言片段存入向量数据库，同时记录时间戳

```json
{
  "content": "项目上线了，熬了两周终于搞完",
  "timestamp": "2026-04-13",
  "emotion": "疲惫但释然",
  "importance": 0.7
}
```

- **检索场景**：用户说"最近工作怎么样"——语义检索召回"项目上线"事件

### 3. 偏好类信息

用户表现出的喜好、习惯、行为倾向。

```
用户：我不喜欢太长的回复，简单说就行
用户：能不能别老问我"你还好吗"，有点烦
用户：我喜欢你用 emoji
```

- **为什么重要**：偏好决定 Agent **怎么回复**，而不仅仅是**回复什么**。属于"程序记忆"——影响的是交互方式
- **存储方式**：规则标记存入用户画像

```json
{
  "reply_style": "简短",
  "avoid_patterns": ["反复追问情绪"],
  "emoji_preference": true
}
```

- **检索场景**：不是用户主动问的，而是 Agent 每次生成回复前**自动加载**，作为 Prompt 的一部分

### 4. 关系类信息

用户提到的人际关系网络。

```
用户：我女朋友叫小雨，她是学设计的
用户：我领导姓王，挺严格的
用户：我有个大学同学叫阿杰，现在在北京
```

- **为什么重要**：用户提到的人名可能反复出现，记不住就会搞混
- **存储方式**：关系图谱或结构化记录

```json
{
  "name": "小雨",
  "relation": "女朋友",
  "attributes": ["学设计的"],
  "first_mentioned": "2026-04-10"
}
```

- **检索场景**：用户说"帮我选个礼物给小雨"——需要知道小雨是谁、学什么的

### 5. 情绪快照

每次对话结束时的情绪状态。

```
第3轮对话：情绪=开心，亲密度+2
第4轮对话：情绪=焦虑，因为工作压力大
第5轮对话：情绪=平静，聊了些轻松话题
```

- **为什么重要**：没有情绪系统的 AI 像客服，有了情绪连续性才像"有感情的伴侣"。用户上次聊完时很焦虑，这次一来就问"今天心情好点了吗"，体验完全不同
- **存储方式**：状态快照存入 KV，极低延迟读取（5-15ms）

```json
{
  "emotion": "焦虑",
  "intensity": 0.7,
  "intimacy": 6.5,
  "trigger": "工作压力",
  "timestamp": "2026-04-13T22:30:00"
}
```

- **检索场景**：每次新对话开始时**自动加载**，不等待用户触发

### 6. 话题上下文

当前正在讨论的话题和未完成的事项。

```
正在讨论：旅行计划（目的地：成都，时间未定）
待确认：用户说"明天再聊这个"
```

- **为什么重要**：保证多轮对话的连贯性。用户说"接着昨天的话题聊"，需要知道昨天聊到哪了
- **存储方式**：对话摘要形式

```json
{
  "topic": "成都旅行计划",
  "resolved": ["目的地选了成都"],
  "pending": ["具体日期", "预算"],
  "last_discussed": "2026-04-12"
}
```

## 收益

- 在有限的 Token 预算内最大化有效信息密度
- 支持跨会话的连续对话体验
- 为不同检索方式（精确查询、语义检索、时序检索）提供结构化的数据基础

## 代价

- 需要额外的信息提取环节（通常用 LLM 做提取，增加一次模型调用）
- 存储成本（向量数据库 + 结构化数据库 + KV）
- 数据一致性维护（双写、更新、过期清理）

## 常见误区

| 误区          | 正确做法                      |
| ----------- | ------------------------- |
| 把所有对话原文都存下来 | 只存提炼后的关键信息                |
| 只用一种存储方式    | 不同类型的信息用不同的存储方式           |
| 只存不说存的重要性   | 标记重要性分数，检索时按重要性加权         |
| 存了就不管了      | 记忆需要过期清理和更新（用户换了工作，旧的要更新） |

## 筛选标准

判断"值不值得存"有三个标准：

```
值得存                           不值得存
─────────────────────────────    ─────────────────────────────
用户透露的个人信息                语气词（嗯、哦、好的）
有时间节点的事件                  重复确认（"你确定吗？"）
情绪变化及触发原因                纯寒暄（"你好"、"在吗"）
用户表达的偏好                    格式化的问答
用户提到的人和关系                已经过时的临时信息
```

> **核心原则：存的是"以后会用到的信息"，扔掉的是"用不到的噪音"。**

## 适用场景

- 所有需要跨会话记忆的 Agent 应用
- AI 伴侣、个人助手、客服机器人、心理咨询 Agent
- 任何需要"记住用户"的对话系统

## 一个真实例子

```
用户：我最近在学 Rust，感觉挺难的

→ 存什么？

  事实：用户正在学 Rust（结构化存储）
  事件：2026-04-14 开始学 Rust，觉得难（向量库 + 时间戳）
  情绪：学习中的挫败感（情绪快照）
  话题：Rust 学习（话题上下文）

→ 不存什么？

  用户在这之前说的"嗯嗯"、"好的"——无信息量
```

# Claude Code 省 Token 指南

> 来源：宝玉（@dotey）2026年4月13日 [推文](https://x.com/dotey/status/2043463546025992494)

## 背景

社区里很多人抱怨 Claude Code 配额烧得太快，Max 用户一周的额度有人两天就用完。很多人的直觉反应是频繁 `/clear` 或不断开新会话，但这往往适得其反。

## 关键机制：提示缓存

- 大模型每次都要从头"读"完整输入，没有记忆
- 提示缓存能把之前的计算结果存下来，读取缓存只要 **十分之一** 的成本
- 缓存只对"前缀"有效，且有存活时间（主智能体 1 小时，子智能体 5 分钟）
- 开新会话 = 缓存全部作废，触发全价重建

## 三个反直觉的省钱策略

1. **缓存还热时，继续聊比开新会话便宜** — 频繁 `/clear` 等于反复为不变的内容付全价写入费
2. **复杂任务一次做对，比来回改三轮更省** — 每多一轮对话，整个上下文都要重新发送一次
3. **长内容给路径，别往对话里贴** — 让 Claude 自己用工具检索，最便宜的 Token 是根本没进上下文的 Token

## 继续聊 vs 开新会话：决策表

### 继续当前会话（满足任一）

- 任务没变，还在调同一个 bug、写同一个模块
- 距离上一条消息不超过 1 小时，缓存还活着
- 上下文里的内容对当前工作仍然有用

### 开新会话（满足任一）

- 任务换了，老会话里的上下文对新任务毫无用处
- 闲置超过 1 小时，缓存大概率已过期
- 上下文被不相关内容塞满，噪音太多

> 一句话：缓存还热、任务没换，继续聊。缓存过期、任务切换、噪音太多，果断重开。

## 1M 上下文窗口：慎用

- 虽然 1M 和 200K 同价，但缓存失效后全量重建代价极高
- 大多数日常会话在 80-120K 就会触发压缩，根本用不到 1M
- 上下文超过 200K 后模型表现明显下降，350K 以上基本靠运气
- 建议设置自动压缩阈值：

```json
{
  "env": {
    "CLAUDE_CODE_AUTO_COMPACT_WINDOW": "200000"
  }
}
```

禁用 1M 上下文：

```json
{
  "env": {
    "CLAUDE_CODE_DISABLE_1M_CONTEXT": "1"
  }
}
```

## 六条操作规则

1. **用 Sonnet 做日常工作** — Opus 消耗 Token 速度约为 Sonnet 的两倍，大多数编码任务 Sonnet 就够了
2. **别在会话中间换模型** — 提示缓存按模型隔离，切模型等于缓存从零开始
3. **精简 CLAUDE.md，控制技能数量** — 控制在 200 行以内，没在用的 MCP 服务记得关掉
4. **命令行优先，MCP 其次** — `gh` 比 GitHub MCP 消耗的 Token 少得多
5. **先花一点 Token 做计划** — 复杂任务先进计划模式，总成本往往更低
6. **用 permissions.deny 限制阅读范围** — 排除 node_modules、构建产物等无关文件：

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./node_modules/**)",
      "Read(./build)"
    ]
  }
}
```

## 委派工作

- **子智能体**：独立上下文，完成后只返回简短摘要，隔离详细输出不污染主会话
- **智能体团队**：每个成员独立实例，计划模式下消耗约为标准会话的 7 倍，慎用
- **Codex 插件**：`claude mcp add codex -- npx -y @openai/codex-plugin-cc`，适合结构化 bug 修复、代码审查、写测试

## 被澄清的误解

- ❌ "上下文超过 256K 之后消耗更快" → 官方：不是真的，实际原因是缓存未命中
- ❌ "模型每次读文件都在检查恶意软件" → Opus 4.6 已移除该提示
- ❌ "自适应思考导致配额异常" → 团队已排除

## 核心思路

> 让缓存尽可能多地被命中，让上下文尽可能少地装无关内容。
>
> "在活跃会话里继续工作"才是默认策略，"开新会话"是特定条件触发的优化操作。

# source-command-vault-rules

Use this skill when the user asks to run the migrated source command `vault-rules`.

## Command Template

# Obsidian Vault 个人规则

这些规则来自 Steph Ango（Obsidian CEO）的实践，已适配本知识库的结构。

## 核心原则

> 拥抱混乱和懒惰，让结构从底向上自然涌现。一致的 style guide 把成百上千个未来决定压缩成一个。

## 规则清单

### 结构规则

1. **不拆分多个 Vault** — 所有知识放在一个库里，通过链接和属性区分，而不是通过物理分隔
2. **不用文件夹来组织** — 文件夹仅用于区分内容性质（参考/剪藏/附件/模板），不用于主题分类
3. **不用非标准 Markdown** — 保持文件在任何 Markdown 编辑器中可读

### 命名规则

4. **类别和标签一律用复数** — `books` 不用 `book`，`movies` 不用 `movie`，避免未来命名犹豫
5. **统一日期格式** — 全部用 `YYYY-MM-DD`，不混用其他格式
6. **文件名即标题** — 不加编号，不加多余后缀，中文命名（除非公认英文术语）

### 链接规则

7. **大量使用内部链接** — 第一次提到就链接，包括尚未创建的笔记（未解析链接 = 未来的面包屑）
8. **属性名跨类别复用** — 如 `genre` 同时适用于书和电影，方便跨类别检索

### 评分规则

9. **7 分制** — 1=邪恶 → 2=糟糕 → 3=差 → 4=凑合 → 5=好 → 6=优秀 → 7=完美。比 5 分有更多区分好体验的粒度，比 10 分少无意义的纠结

### 维护规则

10. **每周一个待办清单** — 不按项目拆分，一个清单管一周
11. **定期随机回顾** — 随机浏览 vault，用 local graph 发现关联，修复格式，创建缺失链接
12. **分形回顾节奏** — 每几天回顾碎片 → 每月回顾 → 每年回顾，形成可缩放的分形网络

## 最重要的一条

> **Don't delegate understanding.** 不要把理解外包出去。维护知识库的过程本身就是理解自己模式的方式。

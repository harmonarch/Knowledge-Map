## 第一阶段：认知与需求建立 (01-03)

> 回答"为什么要做"和"做什么"

### 01、AI 时代的学习心法 ★ 入口锚点
- **核心主张**：把执行交给 AI，把判断留给自己
- **学习分工**：不要平均用力，先抓关键层（架构/抽象/边界），API 细节可交给 AI 补
- **三遍学习法**：① 建立地图 → ② 围绕关键节点深挖 → ③ 边做边学
- **关键判断力**：知道该做什么、不该做什么、哪一层可以妥协、哪一层不能出错

### 02、什么是 AI 智能体开发
- **智能体定义**：让大模型围绕目标，主动召回上下文、调用工具、执行多步动作
- **三大核心能力**：上下文召回、工具调用、多步动作编排
- **定位**：整个专栏的理论起点，从通用角度理解 Agent 开发

### 03、从需求出发，理解 AI 电子伴侣
- **快速原型**：一段 System Prompt 即可体验 AI 伴侣的基本效果
- **从 Demo 到产品的五道鸿沟**：记忆、个性、情感连续性、长期一致性、工程承载
- **RAG 引入**：检索增强生成作为解决 AI "记住你"的核心手段
- **系统定义**：本项目最终要构建的 AI 电子伴侣系统架构

## 第二阶段：核心技术概念 (04-06)

> 回答"核心难题是什么"和"用什么工具"

### 04、内存调度思考 ★ 关键节点
- **核心矛盾**：LLM 无状态 + Token 有上限，但 AI 伴侣需要长期连续上下文
- **调度层的职责**：决定什么进 prompt、什么压缩、什么长期存储、什么丢弃
- **计算机体系结构类比**：L1/L2/L3 缓存 → 短期/中期/长期记忆
- **关键概念**：会话延续性、上下文窗口(context window)、任务优先级与阻塞机制

### 05、LangChain / LangGraph 核心角色
- **分工定位**：
  - LangGraph = 大脑/神经中枢（调度决策、状态管理、记忆调度）
  - LangChain = 器官/肢体（具体功能执行：模型调用、数据处理、工具使用）
- **选型理由**：非必需品，但提供强大的抽象和编排能力
- **类比记忆法**："造人"——大脑指挥，肢体执行

### 06、向量化与语义检索
- **为什么传统搜索不够**：关键词匹配无法理解语义
- **Embedding 原理**：将文本转换为向量表示
- **完整流程**：文本 → 向量化 → 存储 → 语义相似度检索 → 召回
- **技术选型**：CloudFlare Vectorize 作为向量数据库
- **检索质量优化**：提升语义检索召回质量的策略

## 第三阶段：系统设计 (07-10)

> 回答"各模块怎么设计"和"怎么组装成完整系统"

### 07、情绪状态机设计
- **核心差异化**：情绪连续性让 AI 从"客服"变成"有感情的伴侣"
- **有限状态机 (FSM)**：给情绪一个形式化的骨架
- **情绪路由**：用 LangGraph 实现情绪状态的动态路由
- **亲密度系统**：独立维度，与情绪联动
- **衰减机制**：情绪的自然衰减、冲突处理、边界设计

### 08、Prompt 工程深度实践
- **三大致命问题**：所有逻辑混在一起无法维护、无法动态切换、缺乏安全防护
- **分层架构**：将 Prompt 从"一大段文本"升级为独立层次
- **动态组装**：根据情绪状态/场景上下文动态组装
- **Few-shot 示例**：保持角色行为一致性
- **越狱防护**：商业产品必须有系统性安全防护
- **版本管理**：Prompt 作为产品核心资产需要版本迭代

### 09、混合记忆架构设计 ★ 关键节点
- **纯向量检索的局限**：精确事实查询（"我生日几号"）、时间范围查询、多条件组合
- **记忆分类与存储策略**：不同类型记忆需要不同存储方式
- **四条检索通道**：各有适用场景（向量/结构化/时间范围/组合）
- **结果融合策略**：多通道检索结果的融合排序算法
- **双写架构**：记忆写入同时写入不同存储介质

### 10、LangGraph Agent 编排管线 ★★★ 核心合龙节点
- **为什么需要图**：if-else 串联会变成面条代码，DAG 显式建模依赖关系
- **三条路径设计**：
  - **主路径**（95% 请求）：安全检查 → 查询理解 → 并行检索/情绪 → Prompt → LLM → 安全检查 → 返回
  - **拦截路径**：安全未通过时短路返回兜底
  - **异步路径**：返回后异步执行情绪更新 + 记忆写入 + 亲密度计算
- **State 数据总线**：共享状态模式，节点解耦、可替换、调试友好
- **Fork-Join 并行**：记忆检索和情绪读取并行执行
- **同步/异步边界**：判断依据是"输出是否被当前回复需要"
- **节点粒度三原则**：独立可替换、并行机会、调试边界
- **架构意义**：前七篇文章的"合龙"，将所有模块组装为完整处理管线

## 第四阶段：工程架构 (11-15)

> 回答"怎么部署"和"怎么组织代码"

### 11、边缘部署与集群部署对比 ★ 关键节点
- **延迟预算**：TTFB 心理预期 1-2 秒，LLM 推理占 500-2000ms 是硬成本
- **AI 伴侣为什么特别需要边缘部署**：
  - 情感连接对延迟极度敏感
  - 高频短消息模式（非低频长文本）
  - 全球用户无法靠"选区域"解决
  - 流量波峰波谷明显
- **CloudFlare Workers 边缘实践**：边缘推理、边缘存储全家桶、边缘记忆检索

### 12、Hono.js vs FastAPI 技术选型
- **决定性差距**：Hono.js 原生支持 Cloudflare Workers 边缘运行时，FastAPI 不支持
- **技术栈统一**：Hono.js (TypeScript) 与前端 Next.js 统一，减少上下文切换
- **CloudFlare 生态集成**：KV、D1、R2 深度集成
- **结论**：选择 Hono.js，不值得为 FastAPI 放弃边缘部署

### 13、三端架构与 Monorepo 决策
- **三个子系统**：
  - Hono 服务端 → AI 对话中枢神经
  - Next.js 客户端 → 用户界面
  - Next.js 后台管理系统 → 运营控制台
- **Monorepo 五大驱动力**：类型共享、共享业务逻辑、原子提交、统一工具链、与多仓库对比优势
- **工具选型**：Turborepo + Yarn Workspaces

### 14、Monorepo 落地与公共逻辑设计
- **目录结构**：三个应用 (server/web/admin) + 三个共享包 (shared/ui/config)
- **shared 包核心**：
  - Zod Schema → 运行时校验 + TS 类型推导，前后端共享
  - Hono RPC → 端到端类型安全 API 调用
  - 认证鉴权逻辑、业务常量与错误码、工具函数统一导出
- **部署策略**：服务端部署到 Workers，前端部署到 CDN

### 15、Streaming（流式输出架构）
- **必然性**：LLM 生成占总延迟 70-80%，逐 token 生成特性适合流式
- **SSE 协议**：轻量单向推送，原生浏览器支持，适合 LLM 流式场景
- **三层流式管线**：LLM → 服务端 → 前端
- **Hono streamSSE**：自定义事件协议（文本/状态/错误）
- **waitUntil**：Workers 环境下异步后处理（记忆写回等），不阻塞响应
- **断流容错**：三类断流场景、滑动超时检测、优雅降级、幂等性保障

## 第五阶段：可观测性与上线 (16-20)

> 回答"怎么知道系统健康"和"怎么安全上线"

### 16、可观测性 ★ 关键节点
- **AI 调试的非线性特征**：同一症状可能对应 6 种不同根因，且无法可靠复现
- **三大支柱**：Logs（离散事件）、Metrics（系统度量）、Traces（请求链路）
- **Trace + Span 模型**：一次请求 = 一个 Trace，一个操作 = 一个 Span
- **degraded 状态** ★ ：比 error 更危险——"流程还能继续，但已进入次优路径"
  - 不触发报警、表面成功率 100%，但用户体验已悄悄下滑
  - 典型场景：查询理解失败退回全通道、记忆只召回低分结果、Prompt 被 Token 预算裁剪

### 17、LangSmith vs Langfuse vs 自建
- **LangSmith**：LangChain 团队的 SaaS，最强开发调试闭环
- **Langfuse**：开源 LLM observability 平台，可自托管，不绑定生态
- **自建**：边缘环境不建议把第三方平台当生产主账本（延迟/数据主权/成本）
- **务实分工**：
  - 开发调试 → LangSmith
  - 多团队共享 → Langfuse
  - 生产全量归档 → 自建 Tracing
  - 线上排查 → 按需采样到平台

### 18、Workers 下的 Tracing 实现
- **TraceContext**：贯穿整条请求链路的上下文对象，`span()` 方法包装任意异步操作
- **采样策略**（按优先级）：错误全量 > 降级全量 > 超慢全量 > 正常随机采样 5%
- **存储分层**：
  - D1（关系型）→ 完整 Trace，支持条件检索
  - KV（键值）→ 聚合摘要，分钟粒度，24h 过期
- **waitUntil**：Trace 写入不阻塞响应返回

### 19、指标体系与线上排障
- **四类核心指标**：
  - 延迟：TTFB P50/P95/P99、各节点耗时、LLM 首 token 延迟
  - 检索质量：记忆命中率、平均召回条数、Top-1 相似度分布
  - 情绪系统：分类分布、状态切换频率、亲密度增长曲线
  - 成本：Token 消耗、单次对话成本、安全拦截率
- **排障四步法**：找 Trace → 按固定顺序检查节点 → 映射根因 → 定向修复
- **三种典型故障**：记忆"丢失"、情绪"跳变"、延迟突增

### 20、影子模式与灰度验证 ★ 关键节点
- **完整验证链路**：
  ```
  影子模式（确认"不同"）→ A/B 分桶（确认"更好"）→ 渐进发布（确认"可稳定上线"）
  ```
- **影子模式**：新旧节点同时执行，旧节点参与正式回复，新节点只"陪跑"留对比数据
- **A/B 分桶**：确定性哈希保证同一用户始终在同一组，先稳定再随机
- **渐进式发布**：10% → 30% → 50% → 100%，每档留观察窗口
- **自动回滚**：提前定义触发条件（错误率超对照 2 倍、P95 延迟超阈值），不交给人临场判断

## 一句话主线

> **实现交给 AI，理解留给自己；细节可以借力，结构必须亲自掌握。**
>
> 从一个 Prompt Demo 出发 → 理解智能体核心能力 → 设计记忆/情绪/Prompt/编排四大模块 →
> 选择边缘部署 + Monorepo 工程架构 → 建立可观测性 → 影子模式灰度上线。
> 贯穿始终的是：**对系统结构的判断力 > 对 API 细节的熟悉度**。
